# CNT-002: Question Generation — Kimi Batch, Bloom Coverage, Expert Review Pipeline

**Priority:** P0 — blocks all learning sessions (no questions = no sessions)
**Blocked by:** CNT-001 (knowledge graph must exist before questions can reference concept nodes)
**Estimated effort:** 8-12 days (including generation cycles and expert review)
**Contract:** `docs/content-authoring.md` Stages 2-5, `docs/assessment-specification.md`

---

## Context

With the knowledge graph from CNT-001 in place (~2,000 concept nodes for Math 5-unit), this task generates the question bank. The content authoring pipeline (`docs/content-authoring.md` Section 3.2) specifies a corpus-informed generation process: questions are generated **in the style of real Bagrut exams** using the corpus style guide from Stage 0.

The assessment specification (`docs/assessment-specification.md`) defines 8 question types: Multiple Choice, Numeric Answer, Mathematical Expression, True/False with Justification, Ordering/Sequencing, Multi-Part Problem, Proof/Derivation, and Diagram Interpretation. The target is 8-15 questions per concept (mix of types and Bloom's levels), totaling 16,000-30,000 questions for the full Math subject.

Questions are generated by Kimi K2.5 in overnight batch mode, reviewed by Claude Sonnet for quality, validated by SymPy for mathematical correctness, and expert-reviewed for curriculum alignment. The QA pipeline (`docs/content-authoring.md` Section 3.5) enforces minimum coverage: every concept needs >= 5 questions spanning >= 2 Bloom's levels.

## Subtasks

### CNT-002.1: Kimi Batch Generation Pipeline

**Files to create/modify:**
- `scripts/content/generate-questions.py` — batch generation orchestrator
- `scripts/content/prompts/question-generation.txt` — prompt template per question type
- `scripts/content/prompts/question-types/` — one prompt file per question type (8 files)
- `content/math/questions/raw/` — raw LLM-generated questions per topic area
- `config/content/generation-config.yaml` — generation parameters

**Acceptance:**

**Generation process (per `docs/content-authoring.md` Section 3.2):**
- [ ] Input per concept: concept definition + corpus style guide (terminology, question phrasing patterns, distractor patterns, scoring rubrics) + 3-5 real Bagrut questions on this concept as examples
- [ ] Kimi K2.5 batch mode (overnight processing for cost efficiency)
- [ ] Generation prompt instructs: "Generate questions in the exact style of the provided Bagrut examples. Use the same Hebrew terminology. Match the difficulty distribution. Create distractors that reflect real student misconceptions documented in the corpus."
- [ ] Output per concept: 8-15 questions covering at least 3 question types and 2 Bloom's levels

**Question types generated (from `docs/assessment-specification.md`):**

| Type | Target per concept | Evaluation | Bloom's |
|------|-------------------|------------|---------|
| Multiple Choice (4 options) | 3-5 | Deterministic | Remember, Understand, Apply |
| Numeric Answer | 2-3 | Deterministic (tolerance) | Apply, Analyze |
| Mathematical Expression | 1-3 | CAS/SymPy | Apply, Analyze, Create |
| True/False + Justification | 1-2 | Hybrid (deterministic + LLM) | Understand, Analyze, Evaluate |
| Ordering/Sequencing | 0-1 | Deterministic | Understand, Apply |
| Multi-Part Problem | 0-1 | Staged | Apply, Analyze, Evaluate |
| Proof/Derivation | 0-1 | LLM (Sonnet) | Analyze, Evaluate, Create |
| Diagram Interpretation | 0-1 | Deterministic + LLM | Analyze |

**Output schema per question:**
```json

    "Simplify: f'(x) = 2x·cos(x²)"

```

- [ ] Batch processing: 200-400 concepts per day (matching `docs/content-authoring.md` throughput estimate)

- Kimi generates duplicate questions for similar concepts — deduplicate by computing semantic similarity (embedding cosine distance > 0.95 = duplicate)

- Generation cost exceeds budget — monitor cost per concept; if >$0.05/concept, optimize prompts

### CNT-002.2: Claude Sonnet Quality Review + SymPy Validation

**Files to create/modify:**
- `scripts/content/review-questions.py` — automated quality review pipeline
- `scripts/content/prompts/quality-review.txt` — Sonnet review prompt
- `scripts/content/sympy-validator.py` — SymPy mathematical validation
- `content/math/questions/reviewed/` — post-review questions

**Acceptance:**

**Claude Sonnet review (per `docs/content-authoring.md` Section 3.2 step 3):**
- [ ] Each question reviewed by Claude Sonnet for:
  1. Quality: is the question clear, unambiguous, and well-formed?
  2. Bloom's level alignment: does the question actually test at the claimed Bloom's level?
  3. Faithfulness to corpus style: does it "feel like a real Bagrut question"?
  4. Distractor quality: are distractors plausible but clearly wrong? Does each target a specific misconception?
  5. Hebrew language quality: correct grammar, appropriate terminology from the corpus
  6. Difficulty calibration: does the difficulty rating match the question's actual complexity?
- [ ] Sonnet flags questions that fail any criterion with a structured review report:
  ```json
  {
    "question_id": "...",
    "review_status": "approved | flagged | rejected",
    "issues": [
      { "criterion": "bloom_alignment", "severity": "warn", "detail": "Claims 'analyze' but tests 'apply'" },
      { "criterion": "distractor_quality", "severity": "error", "detail": "Option C is obviously wrong" }
    ],
    "suggested_fixes": "..."
  }
  ```
- [ ] Review pass rate target: >70% approved on first pass (per `docs/content-authoring.md` rejection rate target of <30%)

**SymPy validation (per `docs/assessment-specification.md` Section 1.3):**
- [ ] All numeric answer questions: verify correct answer is mathematically valid
- [ ] All expression questions: verify correct answer matches reference via `sympy.simplify(student - reference) == 0`
- [ ] All MCQ with mathematical content: verify the correct option is actually correct and distractors are actually wrong
- [ ] Multi-part problems: verify each part's answer is consistent with previous parts
- [ ] Timeout: 2 seconds per validation (same as runtime CAS evaluator)
- [ ] SymPy validation results logged: `{question_id, valid: bool, computation_time_ms, error_details}`

**Test:**
```bash
# Run quality review on test batch
python3 scripts/content/review-questions.py \
  --input content/math/questions/raw/test_batch/ \
  --output content/math/questions/reviewed/test_batch/ \
  --model claude-sonnet-4-6

# Check review results
python3 -c "
import json, os
from collections import Counter
statuses = Counter()
for f in os.listdir('content/math/questions/reviewed/test_batch/'):
    if f.endswith('.json'):
        q = json.load(open(f'content/math/questions/reviewed/test_batch/{f}'))
        statuses[q.get('review_status', 'unknown')] += 1
total = sum(statuses.values())
approved = statuses.get('approved', 0)
rate = approved / total * 100 if total > 0 else 0
print(f'Review results: {dict(statuses)}')
print(f'Approval rate: {rate:.1f}% (target: >70%)')
if rate >= 70: print('PASS')
else: print('WARN: approval rate below target')
"

# Run SymPy validation
python3 scripts/content/sympy-validator.py \
  --input content/math/questions/reviewed/test_batch/ \
  --output content/math/questions/validated/

# Check SymPy results
python3 -c "
import json, os
valid = invalid = 0
for f in os.listdir('content/math/questions/validated/'):
    if f.endswith('.json'):
        result = json.load(open(f'content/math/questions/validated/{f}'))
        if result.get('sympy_valid'): valid += 1
        else: invalid += 1
print(f'SymPy validation: {valid} valid, {invalid} invalid')
if invalid > 0: print(f'WARN: {invalid} questions have mathematical errors')
else: print('PASS: all questions mathematically valid')
"
```

**Edge cases:**
- Sonnet review cost at scale (30,000 questions * ~500 tokens/review) — ~$45; acceptable
- SymPy timeout on complex integrals — fall back to numeric sampling at 5 random points (per `docs/assessment-specification.md`)
- Sonnet flags a question as "not Bagrut style" but the concept is new to the syllabus (no Bagrut examples exist) — relax corpus style requirement for concepts with `bagrut_frequency: none`
- SymPy parsing fails on Hebrew-annotated LaTeX — strip Hebrew text before SymPy parsing; validate Hebrew separately

### CNT-002.3: Coverage Analysis and Gap Filling

**Files to create/modify:**
- `scripts/content/coverage-analyzer.py` — analyzes question bank coverage
- `scripts/content/gap-filler.py` — generates missing questions for under-covered concepts
- `content/math/questions/coverage-report.json` — coverage analysis output

**Acceptance:**

**Coverage requirements (per `docs/content-authoring.md` Section 3.5):**
- [ ] Every concept has >= 5 questions
- [ ] Every concept has questions spanning >= 2 Bloom's levels
- [ ] Every concept has questions spanning >= 2 question types
- [ ] Difficulty distribution per concept spans at least 3 difficulty levels (1-5 scale)
- [ ] High-frequency Bagrut concepts (`bagrut_frequency: high`) have >= 10 questions
- [ ] At least one question per concept has full solution steps (for hint escalation)

**Bloom's coverage per concept:**
- [ ] Must have at least: 1 Remember/Understand + 1 Apply/Analyze
- [ ] Concepts at `bloom_level: evaluate` or `create` must have at least 1 higher-order question

**Coverage report:**

- [ ] Gap filler runs targeted generation only for under-covered concepts
- [ ] Gap generation uses the same pipeline (Kimi batch -> Sonnet review -> SymPy validation)
- [ ] Coverage analyzer runs automatically after every generation batch

**Test:**

**Edge cases:**
- Concept has 5 questions but all are MCQ — coverage analyzer catches missing type diversity; gap filler generates non-MCQ types
- High-frequency concept with only 6 questions — gap filler prioritizes high-frequency concepts first
- Coverage drops after SymPy invalidates questions — re-run coverage after validation, not before
- Some concepts are inherently difficult to write diverse questions for (e.g., "definition of limit") — accept lower type diversity for `bloom_level: remember` concepts

### CNT-002.4: Expert Review Pipeline and Publication

**Files to create/modify:**
- `scripts/content/prepare-review-queue.py` — prepares review queue sorted by priority
- `content/math/questions/review-queue.json` — prioritized queue for domain expert
- `scripts/content/publish-questions.py` — publishes reviewed questions to PostgreSQL + S3
- `content/math/questions/published/` — final published question bank

**Acceptance:**

**Review queue priority (per `docs/content-authoring.md` Section 3.4):**
1. Questions for concepts with wrong prerequisites (cascading failure risk)
2. Questions with low generation confidence (<0.8)
3. Questions flagged by Sonnet quality review
4. Questions with SymPy validation warnings (borderline cases)
5. Questions for high-frequency Bagrut concepts (most student impact)

**Expert review throughput:**
- [ ] Expert reviews 100-150 concepts/week (per `docs/content-authoring.md`)
- [ ] At 10 questions/concept, that's 1,000-1,500 questions/week
- [ ] Full Math subject (2,000 concepts): ~15 weeks with 1 expert, ~8 weeks with 2
- [ ] High-confidence items (>0.95) eligible for bulk approval

**Expert rejection handling (per `docs/content-authoring.md` Section 3.4):**
- [ ] Rejection rate tracked per-stage and per-topic-area
- [ ] Target: <30% rejection rate
- [ ] If rejection rate >40% for a concept cluster: flag for prompt engineering review
- [ ] If rejection rate >50% for an entire subject: pipeline pauses; root cause analysis
- [ ] Rejected items tagged with reason: `factual_error`, `wrong_difficulty`, `poor_hebrew`, `wrong_prerequisite`, `wrong_bloom_level`
- [ ] Rejection reasons aggregated weekly to identify systematic LLM weaknesses
- [ ] Accepted-with-edits items used as few-shot examples for improving generation prompts

**Publication:**
- [ ] Published questions stored in PostgreSQL (Marten document store) for runtime access
- [ ] Published questions also exported to S3 as versioned JSON artifacts: `s3://cena-content/math/questions/v1.0.0/`
- [ ] Publication emits `ContentPublished` event to NATS (`cena.content.events.ContentPublished`)
- [ ] Question bank versioned: `math_5u_questions_v1.0.0`
- [ ] Rollback: previous version can be restored from S3

**Edge cases:**
- Expert is unavailable for weeks — high-confidence questions (>0.95) can be bulk-published with a lower review bar; flag as "auto-approved, pending review"
- Expert edits change the correct answer — re-run SymPy validation after edit
- Publication fails mid-way — S3 upload is atomic per file; PostgreSQL uses a transaction; partial publication leaves the system in a valid state (old version + some new questions)
- Two experts disagree on a question — escalate to education advisor; document the disagreement for prompt improvement

## Rollback Criteria

If this task fails or introduces instability:
- Question generation can be re-run: Kimi batch jobs are idempotent; raw output is versioned in S3
- Published question bank can be rolled back to previous version via S3 versioning
- Expert review edits are tracked in the review tool; no work is lost
- If SymPy validation has false negatives (incorrectly marks valid questions as invalid): disable SymPy validation and rely on expert review only
