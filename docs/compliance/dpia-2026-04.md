## 1. The Need for a DPIA

### 1.1 Why this DPIA is required

GDPR Article 35(1) requires a DPIA where processing is "likely to result in a
high risk to the rights and freedoms of natural persons." The Article 29 Working
Party (WP29) Guidelines on DPIAs (WP 248 rev.01) identify nine criteria that
indicate high risk. Cena's processing triggers **five** of the nine:

| WP29 Criterion | Triggered? | Cena Evidence |
|---|---|---|
| 1. Evaluation or scoring | YES | Elo rating, Bayesian Knowledge Tracing mastery prediction, adaptive difficulty calibration |
| 2. Automated decision-making with legal/significant effect | YES | Adaptive question selection directly affects educational progression and reported outcomes |
| 3. Systematic monitoring | YES | Per-session behavioral analytics (FocusAnalyticsService), time-on-task tracking, engagement signals |
| 4. Sensitive data or data of highly personal nature | YES | Children's educational performance data is highly personal per ICO guidance |
| 5. Data processed on a large scale | PARTIAL | Multi-tenant platform serving multiple schools; scale expected to grow |
| 6. Datasets that have been matched or combined | YES | Student profile + Elo rating + mastery state + tutor conversation + analytics events combined per student |
| 7. Data concerning vulnerable data subjects | YES | **Children / minors** -- the primary user base |
| 8. Innovative use or applying new technology | YES | AI-powered tutoring (Anthropic Claude) interacting directly with minors |
| 9. Processing that prevents exercising a right or using a service/contract | NO | N/A |

Triggering two or more criteria makes a DPIA **mandatory** (WP29, p. 11). Cena
triggers five to six criteria. A DPIA is therefore legally required before
processing begins.

### 1.2 ICO Children's Code relevance

The ICO Age Appropriate Design Code (Children's Code) Standards 1, 5, and 12
are directly engaged:

- **Standard 1 (Best interests of the child)**: The DPIA exercise is the primary
  mechanism to demonstrate that processing serves children's best interests.
- **Standard 5 (Detrimental use of data)**: Profiling children via Elo ratings
  and mastery predictions must be assessed for detrimental use.
- **Standard 12 (Profiling)**: Profiling of children must be OFF by default.
  Cena profiles from the moment of registration (EloRating=1500 default in
  `StudentProfileSnapshot.cs:41`). This DPIA must assess the risk and identify
  mitigations.

### 1.3 Additional framework obligations

- **COPPA** (16 CFR Part 312): US children's privacy; requires notice and
  verifiable parental consent for under-13s.
- **FERPA** (34 CFR Part 99): US educational records privacy; restricts
  disclosure of educational records.
- **Israel Protection of Privacy Law (PPL)**: Israeli data protection regime;
  Amendment 13 (2024) introduced enhanced minor protections.

## 2. Description of the Processing

### 2.1 System overview

Cena is a multi-tenant adaptive learning platform that serves school-age
students (primary and secondary). It provides:

- Curriculum-aligned practice questions adapted to each learner's ability
- AI-powered tutoring via a conversational interface (Anthropic Claude)
- Progress dashboards for students, teachers, and (future) parents
- Gamification (leaderboards, streaks, challenges)
- Administrative analytics for schools

### 2.2 Nature of the processing

| Processing Activity | Personal Data Involved | Data Subjects | Legal Basis (GDPR Art 6) |
|---|---|---|---|
| **Account authentication** | Email, display name, hashed password | Students (including minors), teachers, admins | Art 6(1)(b) Contract |
| **Session management** | Session tokens, device metadata, IP (anonymized) | Students | Art 6(1)(b) Contract |
| **Adaptive learning recommendations** | Elo rating, mastery state (BKT), question response history, time-on-task | Students | Art 6(1)(a) Consent (parental for minors) |
| **Peer comparison** | Anonymized aggregate performance vs. peers | Students | Art 6(1)(a) Consent |
| **Leaderboard display** | Display name, points, streak count | Students | Art 6(1)(a) Consent |
| **Social features** | Posts, comments, display name | Students | Art 6(1)(a) Consent |
| **AI tutoring (Anthropic Claude)** | Free-text conversation messages, student context (subject, mastery level) | Students | Art 6(1)(a) Consent |
| **Behavioral analytics** | Focus tracking events, engagement signals, session timing | Students | Art 6(1)(a) Consent |
| **Cross-tenant benchmarking** | Anonymized aggregate metrics across schools | Students (aggregated) | Art 6(1)(a) Consent |
| **Administrative analytics** | Per-student and aggregate academic performance | Students, teachers | Art 6(1)(f) Legitimate interest |

### 2.3 Processing purposes (mapped to codebase)

The codebase defines 10 granular processing purposes in

1. `AccountAuth` -- Contract basis, always required
2. `SessionContinuity` -- Contract basis, always required
3. `AdaptiveRecommendation` -- Consent basis, minor default: ON
4. `PeerComparison` -- Consent basis, minor default: OFF
5. `LeaderboardDisplay` -- Consent basis, minor default: OFF
6. `SocialFeatures` -- Consent basis, minor default: OFF
7. `ThirdPartyAi` -- Consent basis, minor default: OFF
8. `BehavioralAnalytics` -- Consent basis, minor default: OFF
9. `CrossTenantBenchmarking` -- Consent basis, minor default: OFF
10. `MarketingNudges` -- Consent basis, minor default: OFF

### 2.4 Scope of data subjects

- **Primary**: Students aged 6--18 (minors)
- **Secondary**: Teachers and school administrators (adults)
- **Indirect**: Parents/guardians (contact data for consent flows)

## 4. Necessity and Proportionality

### 4.1 Why this processing is necessary

Cena's core educational purpose requires processing student performance data to
deliver adaptive learning. The specific necessity for each processing activity:

| Processing | Necessity | Could the purpose be achieved with less data? |
|---|---|---|
| **Elo rating** | Required to calibrate question difficulty to the learner's level. Without it, questions would be random or fixed-sequence, reducing educational efficacy. | The Elo algorithm requires a running score per student. Minimum data: one numeric rating + response outcome per question. This is already minimal. |
| **BKT mastery prediction** | Required to determine when a student has mastered a topic and can progress. Prevents unnecessary repetition and enables targeted remediation. | BKT requires per-topic attempt/success counts. Already minimal for the purpose. |
| **AI tutoring** | Provides real-time personalized explanations when a student is stuck. Addresses the gap between one teacher and many students. | Conversation text is required for the AI to respond. PII scrubbing before transmission to Anthropic is a proportionality measure (see Mitigations). |
| **Focus analytics** | Identifies when a student is disengaged or struggling, enabling teacher intervention. | Session-level timing and engagement signals are the minimum needed. Per-keystroke or webcam tracking would be disproportionate and is not used. |
| **Cross-tenant benchmarking** | Helps schools understand relative performance. | Only anonymized aggregates cross tenant boundaries. Individual-level data stays within the tenant. |

### 4.2 Proportionality measures already in place

- **Data minimization**: Only email, display name, and date of birth (when
  implemented) are collected. No home address, phone number, or biometric data.
- **Purpose limitation**: 10 granular processing purposes defined in code, each
  with an explicit lawful basis.
- **Minor-specific defaults**: 7 of 10 processing purposes default to OFF for
  minors (per `[MinorDefault(false)]` attribute in code).
- **Retention limits**: AI tutor conversations auto-purge after 12 months.
  IP addresses anonymized within 90 days.
- **No marketing to children**: `MarketingNudges` defaults to OFF for all users
  and is consent-gated.

### 4.3 Proportionality gaps identified

- **Elo rating assigned at registration** (`StudentProfileSnapshot.cs:41`,
  `EloRating = 1500`): Profiling begins immediately upon account creation,
  before any consent is recorded. This contravenes ICO Children's Code Std 12
  (profiling OFF by default). **Tracked as FIND-privacy-009 / mitigation M-5.**
- **AdaptiveRecommendation minor default is ON**: This is the core product
  purpose but means profiling is not fully OFF by default for minors. Requires
  parental consent gate before activation. **Tracked as FIND-privacy-001.**

## 5. Risks to Individuals

### Risk register

| ID | Risk | Likelihood | Impact | Overall | Data Subjects Affected |
|---|---|---|---|---|---|
| R-1 | **Profiling causes educational harm**: Elo/BKT mislabeling locks a child into inappropriately easy or hard content, affecting their educational outcomes and self-perception | Medium | High | **HIGH** | Students (minors) |
| R-2 | **AI tutor produces inappropriate content**: Anthropic Claude generates content that is harmful, misleading, or age-inappropriate for a child | Low | High | **MEDIUM** | Students (minors) |
| R-3 | **AI tutor elicits or processes sensitive disclosures**: A child discloses abuse, self-harm, or other safeguarding concerns via the tutor; data is transmitted to Anthropic without appropriate handling | Medium | High | **HIGH** | Students (minors) |
| R-4 | **Cross-border data transfer to Anthropic (US)**: Student conversation data is transferred to US servers; potential access by US authorities under FISA 702 / EO 12333 | Medium | Medium | **MEDIUM** | Students (minors), all users |
| R-5 | **Data breach exposing children's educational records**: Unauthorized access to PostgreSQL exposes Elo ratings, mastery states, session data, and conversation history | Low | High | **MEDIUM** | Students (minors) |
| R-6 | **Behavioral profiling enables surveillance**: FocusAnalytics data (time-on-task, engagement signals) could be misused for surveillance of children's behavior beyond educational purpose | Low | High | **MEDIUM** | Students (minors) |
| R-7 | **Social visibility causes peer pressure or bullying**: Leaderboards, peer comparison, and social features expose a child's performance to peers, causing distress | Medium | Medium | **MEDIUM** | Students (minors) |
| R-8 | **Consent not validly obtained for minors**: Registration flow does not collect date of birth or parental consent; processing of children's data may lack lawful basis entirely | High | High | **HIGH** | Students (minors) |
| R-9 | **Right to erasure not effectively exercisable**: RightToErasureService is incomplete (only deletes consent records and access logs, not the event stream or profile snapshot) | High | Medium | **HIGH** | Students (minors), all users |
| R-10 | **Lack of transparency**: No privacy notice at point of collection; children's privacy notice not surfaced in the registration flow | High | Medium | **HIGH** | Students (minors) |
| R-11 | **No parental visibility or control**: No parent-facing dashboard; parents cannot review or delete their child's data without contacting an admin | High | Medium | **HIGH** | Parents/guardians |
| R-12 | **Retention beyond necessity**: Some data categories (learning events, behavioral analytics) have no defined auto-purge; retained for account lifetime without periodic review | Medium | Medium | **MEDIUM** | Students (minors) |

| M-4 | R-4 | **PII scrubbing before Anthropic transmission**: Strip or pseudonymize personally identifiable information from conversation context before sending to Anthropic API | PLANNED | FIND-privacy-008 |
| M-5 | R-1, R-6 | **Profiling OFF by default for minors**: Defer Elo rating initialization and adaptive recommendation activation until parental consent is confirmed; implement ICO Std 12 toggle | PLANNED | FIND-privacy-009 |
| M-6 | R-9 | **Complete right-to-erasure implementation**: Extend RightToErasureService to delete the full event stream, StudentProfileSnapshot, tutor conversations, and all associated analytics events | PLANNED | FIND-privacy-005 |
| M-7 | R-7 | **High-privacy defaults for social features**: Set `ProfileVisibility=private`, `ShowProgressToClass=false`, `AllowPeerComparison=false` as defaults for all users; require explicit opt-in for minors | PLANNED | FIND-privacy-010 |
| M-8 | R-5 | **Database encryption and access controls**: Encrypt PostgreSQL at rest and in transit; implement row-level security for tenant isolation; audit logging for all data access | PARTIAL | Existing infra + TENANCY-P1 |
| M-9 | R-4 | **Cross-border transfer safeguards**: Execute Standard Contractual Clauses (SCCs) with Anthropic; conduct Transfer Impact Assessment (TIA); implement supplementary technical measures (encryption in transit, PII scrubbing) | PLANNED | FIND-privacy-008 |
| M-10 | R-12 | **Automated retention enforcement**: Implement automated data purge for learning events and analytics beyond the defined retention period; extend ErasureWorker to handle periodic retention sweeps | PLANNED | FIND-privacy-004 |
| M-11 | R-11 | **Parent dashboard**: Build parent-facing interface allowing parents to review their child's data, exercise access/deletion rights, and manage consent without contacting admin | PLANNED | FIND-privacy-001 |
| M-12 | R-1 | **Profiling accuracy monitoring**: Implement ongoing monitoring of Elo/BKT calibration accuracy; alert when a student's profile diverges significantly from observed performance; provide teacher override mechanism | PLANNED | (future task) |
| M-13 | R-8 | **Consent enforcement middleware**: Activate the existing ConsentEnforcementMiddleware to block processing endpoints when required consent is not recorded | PLANNED | FIND-privacy-007 |

### Residual risks after all mitigations

| Risk | Residual Level | Rationale |
|---|---|---|
| R-1 (Profiling harm) | LOW | With parental consent, profiling toggle, and accuracy monitoring, risk is reduced to acceptable levels for educational use |
| R-2 (Inappropriate AI content) | LOW | Content filtering + safeguarding detector + parental consent for ThirdPartyAi reduces exposure |
| R-3 (Sensitive disclosures) | LOW-MEDIUM | Safeguarding detector will catch many cases but cannot guarantee 100% detection; human escalation path is critical |

| R-6 (Behavioral surveillance) | LOW | Purpose limitation + consent gating + minor defaults OFF restricts analytics to educational purpose |
| R-7 (Social pressure) | LOW | High-privacy defaults + opt-in for minors eliminates involuntary exposure |
| R-8 (Invalid consent) | LOW | Age gate + parental consent flow + consent enforcement middleware closes the gap |
| R-9 (Incomplete erasure) | LOW | Complete erasure implementation removes all personal data |
| R-10 (Lack of transparency) | LOW | Privacy notices at point of collection + children's notice satisfies Art 13 |
| R-11 (No parental control) | LOW | Parent dashboard provides direct access and control |
| R-12 (Over-retention) | LOW | Automated retention enforcement ensures data is not kept beyond necessity |

## 9. Review Schedule

### 9.1 Review triggers

This DPIA must be reviewed and updated:

1. **Quarterly**: Every 3 months from the date of this assessment (next review:
   2026-07-12).
2. **On material change**: Any of the following triggers an immediate review:
   - New processing purpose added to `ProcessingPurpose` enum
   - New third-party processor engaged (e.g., analytics provider, new AI model)
   - Change in data subjects (e.g., extending to users under 6 or over 18)
   - Change in data storage location or cross-border transfer arrangement
   - Security incident involving personal data
   - Significant change to Elo/BKT algorithm affecting profiling outcomes
   - Expansion to new jurisdiction with different age-of-consent thresholds
   - Change in Anthropic's data processing terms or model capabilities
3. **Annually**: Full re-assessment regardless of changes (next: 2027-04-12).

### 9.3 DPIA naming convention

DPIA documents follow the naming convention `dpia-YYYY-QN.md` where:
- `YYYY` = calendar year
- `QN` = quarter number (Q1 = Jan-Mar, Q2 = Apr-Jun, Q3 = Jul-Sep, Q4 = Oct-Dec)

The CI gate (`scripts/ci-dpia-check.mjs`) asserts that a DPIA file matching the
current quarter exists in `docs/compliance/`. This prevents the assessment from
lapsing.
