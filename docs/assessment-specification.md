## 1. Question Taxonomy

Cena supports eight question types across all Bagrut STEM subjects (Mathematics, Physics, Chemistry, Biology, Computer Science). Each type is defined by its input format, evaluation method, partial credit rules, hint escalation strategy, Bloom's taxonomy coverage, and primary subject affinity.

---

### 1.1 Multiple Choice (4 Options)

| Attribute | Value |
|-----------|-------|
| **Format** | Stem + 4 answer options (A–D), exactly one correct |
| **Input modality** | Tap/select one option |
| **Input UI component** | `<MCQSelector>` — vertical radio-button list with RTL label alignment; option randomization on each attempt |
| **Evaluation pipeline** | **Deterministic** — exact match against correct option ID |
| **Partial credit** | None (0 or 1). Future: distractor analysis may award 0.25 for "close" distractors if A/B test shows benefit |
| **Hint escalation** | L1 Nudge: "Eliminate one option — which answer is definitely wrong?" / L2 Scaffold: Eliminate two incorrect options, explain why / L3 Near-answer: "The answer relates to [concept]. Think about [specific relationship]." |
| **Bloom's levels** | Remember, Understand, Apply |
| **Primary subjects** | All subjects — workhorse question type |

**Distractor design principles:**
- Each distractor targets a specific misconception or common error (documented in question metadata)
- One distractor is the "obvious wrong" (eliminable with basic knowledge)
- One distractor is the "near miss" (tests precise understanding)
- One distractor is the "common misconception" (addresses known error patterns from the concept's `common_misconceptions` field)
- Distractors are generated by Kimi K2.5 and reviewed by domain experts during content authoring

### 1.2 Numeric Answer

| Attribute | Value |
|-----------|-------|
| **Format** | Stem poses a computation or measurement; student enters a number |
| **Input modality** | Numeric keypad with decimal point and negative sign |
| **Input UI component** | `<NumericInput>` — numeric soft keyboard, unit selector dropdown (optional), significant figures indicator |
| **Evaluation pipeline** | **Deterministic** — exact match with configurable tolerance (default ±0.01, per-question override). Unit validation: if unit is required, answer must include correct unit |
| **Partial credit** | 0.5 if numeric value is correct but unit is wrong or missing (when unit is required). 0 otherwise |
| **Hint escalation** | L1 Nudge: "Check your units — what unit should the answer be in?" / L2 Scaffold: "Start by writing down the formula. What values do you plug in?" / L3 Near-answer: "The answer is between [range]. Try recalculating step [N]." |
| **Bloom's levels** | Apply, Analyze |
| **Primary subjects** | Mathematics, Physics, Chemistry |

**Tolerance configuration:**
```json
{
  "tolerance_type": "absolute",    // "absolute" | "relative" | "significant_figures"
  "tolerance_value": 0.01,         // ±0.01 for absolute
  "unit_required": true,
  "accepted_units": ["m/s", "m·s⁻¹"],
  "significant_figures": null       // null = not checked, 3 = must match 3 sig figs
}
```

### 1.3 Mathematical Expression

| Attribute | Value |
|-----------|-------|
| **Format** | Stem asks student to derive, simplify, or solve; student enters a mathematical expression |
| **Input modality** | LaTeX-style math input with visual rendering |
| **Input UI component** | `<MathInput>` — integrated math keyboard (fractions, exponents, roots, trig, Greek letters) with live LaTeX preview. Based on MathQuill or similar React Native math editor |
| **Evaluation pipeline** | **CAS (SymPy service)** — Python sidecar parses student expression and reference answer, tests symbolic equivalence via `sympy.simplify(student - reference) == 0`. Falls back to numeric sampling (evaluate at 5 random points) if symbolic comparison times out (>2s) |
| **Partial credit** | 0.5 if expression is algebraically equivalent after fixing a single sign error (detected by CAS). 0.25 if expression matches the correct intermediate step (requires intermediate steps defined in question metadata) |
| **Hint escalation** | L1 Nudge: "What rule or formula applies here?" / L2 Scaffold: "The first step is [intermediate result]. What comes next?" / L3 Near-answer: "Your answer should look like [form]. You're close — check the [denominator/exponent/coefficient]." |
| **Bloom's levels** | Apply, Analyze, Create |
| **Primary subjects** | Mathematics, Physics |

**CAS evaluator details:**
- SymPy runs as a Python sidecar service behind the LLM ACL's FastAPI layer
- Input parsing: student expression string → SymPy `sympify()` with safe parsing (no `eval`)
- Equivalence check: `simplify(student_expr - reference_expr) == 0`
- Timeout: 2 seconds per equivalence check; if exceeded, fall back to numerical evaluation at 5 random points
- Numerical tolerance for fallback: relative error < 1e-6 at all sample points
- Example: student writes `3/(3x+1)`, reference is `3(3x+1)^{-1}` → SymPy confirms equivalence

### 1.4 True/False with Justification

| Attribute | Value |
|-----------|-------|
| **Format** | Statement presented; student selects True or False, then writes a justification |
| **Input modality** | Binary toggle + free-text input |
| **Input UI component** | `<TFJustification>` — True/False toggle buttons + multi-line text area with 200-character minimum |
| **Evaluation pipeline** | **Hybrid** — T/F part: deterministic exact match (0.4 of total score). Justification part: LLM evaluation via Kimi K2.5 for classification against rubric (0.6 of total score) |
| **Partial credit** | T/F correct + justification correct = 1.0 / T/F correct + justification weak = 0.6 / T/F correct + justification wrong = 0.4 / T/F wrong = 0.0 (justification not evaluated if binary part is wrong) |
| **Hint escalation** | L1 Nudge: "Think about what would happen if [counter-scenario]." / L2 Scaffold: "Consider the definition of [concept]. Does the statement satisfy all conditions?" / L3 Near-answer: "The statement is [True/False]. Your justification should mention [key term]." |
| **Bloom's levels** | Understand, Analyze, Evaluate |
| **Primary subjects** | Biology, Chemistry, Physics |

**LLM justification evaluation:**
- Kimi K2.5 receives: `{ "statement": "...", "correct_answer": true/false, "student_justification": "...", "rubric_keywords": ["keyword1", "keyword2"], "rubric_criteria": "..." }`
- Kimi classifies justification as: `correct`, `partially_correct`, `incorrect`
- No PII sent — student identified only by opaque session ID
- Fallback: if Kimi unavailable, justification receives full credit if T/F is correct (graceful degradation)

### 1.5 Ordering/Sequencing

| Attribute | Value |
|-----------|-------|
| **Format** | 4–8 items presented in random order; student arranges them in the correct sequence |
| **Input modality** | Drag-and-drop reordering |
| **Input UI component** | `<DragSequence>` — vertical sortable list with drag handles, haptic feedback on reorder, accessibility support via up/down buttons |
| **Evaluation pipeline** | **Deterministic** — compare student sequence against correct sequence |
| **Partial credit** | Kendall tau distance: `score = 1 - (2 * inversions) / (n * (n-1))` where n is item count and inversions is the number of pairwise swaps needed. Minimum score: 0 |
| **Hint escalation** | L1 Nudge: "The first step is correct. Focus on what comes after step [N]." / L2 Scaffold: "Steps [A] and [B] are in the wrong order. Think about which must come first and why." / L3 Near-answer: Reveal the first 2 items in correct position, student orders the rest |
| **Bloom's levels** | Understand, Analyze |
| **Primary subjects** | Chemistry (reaction steps), Biology (processes), Mathematics (proof steps), Physics (problem-solving procedures) |

### 1.6 Fill-in-the-Blank

| Attribute | Value |
|-----------|-------|
| **Format** | Text passage with 1–3 blanks; student fills each blank with a word or short phrase |
| **Input modality** | Inline text input fields within passage |
| **Input UI component** | `<FillBlank>` — rendered passage with inline `<TextInput>` fields at blank positions, character limit per blank (configurable, default 50 chars) |
| **Evaluation pipeline** | **Deterministic** — keyword matching with synonym acceptance. Each blank has a list of accepted answers (exact strings + synonyms). Case-insensitive, whitespace-normalized |
| **Partial credit** | Per-blank scoring: each blank is worth `1/num_blanks` of total. Blank is either correct (1) or incorrect (0). No partial within a single blank |
| **Hint escalation** | L1 Nudge: "The answer to blank [N] starts with the letter [X]." / L2 Scaffold: "Think about the definition of [related concept]. The blank asks for a [noun/verb/term]." / L3 Near-answer: "The answer is [first 3 characters]..." |
| **Bloom's levels** | Remember, Understand |
| **Primary subjects** | Biology (terminology), Chemistry (nomenclature), Computer Science (syntax) |

**Synonym acceptance:**
```json
{
  "blank_1": {
    "canonical": "mitochondria",
    "accepted": ["mitochondria", "mitochondrion", "המיטוכונדריה", "מיטוכונדריה"],
    "rejected_with_feedback": {
      "nucleus": "Close — that's the control center, not the energy producer."
    }
  }
}
```

### 1.7 Diagram Labeling

| Attribute | Value |
|-----------|-------|
| **Format** | SVG/image diagram with numbered hotspot regions; student taps each region and selects/types the correct label |
| **Input modality** | Tap on diagram region → select label from bank OR type label |
| **Input UI component** | `<DiagramLabeler>` — zoomable SVG/image canvas with defined tap regions (polygonal hotspots), label bank sidebar (drag or tap to assign), pinch-to-zoom on mobile |
| **Evaluation pipeline** | **Deterministic** — each region has a correct label ID; compare student assignments against answer key |
| **Partial credit** | Per-region scoring: `score = correct_labels / total_labels`. Each label is binary (correct/incorrect) |
| **Hint escalation** | L1 Nudge: "Region [N] is incorrectly labeled. Look at its position relative to [landmark]." / L2 Scaffold: Highlight 2 correctly-labeled regions as anchors, explain their relationship to the unlabeled region / L3 Near-answer: Reveal labels for all but 2 regions, student labels remaining |
| **Bloom's levels** | Remember, Understand, Apply |
| **Primary subjects** | Biology (cell diagrams, organ systems), Physics (circuit diagrams, force diagrams), Chemistry (molecular structures, apparatus) |

**Diagram specification format:**

    {
      "region_id": "r1",
      "polygon": [[120,45], [180,45], [180,90], [120,90]],
      "correct_label": "nucleus",
      "accepted_labels": ["nucleus", "גרעין"]

### 1.8 Free-Text Explanation

| Attribute | Value |
|-----------|-------|
| **Format** | Open prompt: "Explain why...", "Describe the relationship between...", "What would happen if..." |
| **Input modality** | Multi-line free text |
| **Input UI component** | `<FreeTextExplainer>` — multi-line text area with 500-character minimum, word count indicator, optional voice-to-text input for accessibility |
| **Evaluation pipeline** | **LLM (Claude Sonnet 4.6)** — evaluates conceptual accuracy, completeness, and reasoning quality against a rubric. Returns structured score + feedback |
| **Partial credit** | Rubric-based 0–1 continuous score. Sonnet evaluates against 3–5 rubric criteria, each weighted. Example: conceptual accuracy (40%), completeness (30%), reasoning clarity (20%), use of technical terms (10%) |
| **Hint escalation** | L1 Nudge: "Start by defining [key concept]. Then explain how it connects to [related concept]." / L2 Scaffold: "Your explanation should cover these points: [bullet list of 2-3 key ideas]." / L3 Near-answer: Provide a partial model answer with gaps: "[Concept A] causes [___] because [Concept B] affects [___]." |
| **Bloom's levels** | Understand, Analyze, Evaluate, Create |
| **Primary subjects** | All subjects — used for Feynman technique assessment ("explain it like you're teaching someone") |

**LLM evaluation prompt structure:**
```
You are evaluating a student's explanation for conceptual accuracy.

Question: {question_text}
Reference answer: {reference_answer}
Rubric: {rubric_criteria}
Student answer: {student_answer}

Evaluate on:
1. Conceptual accuracy (0-1): Are the core concepts correct?
2. Completeness (0-1): Are all key points addressed?
3. Reasoning clarity (0-1): Is the logic sound and clearly expressed?
4. Technical vocabulary (0-1): Are domain terms used correctly?

Return JSON: { "scores": {...}, "weighted_total": 0.XX, "feedback": "..." }
```

- Evaluated by Claude Sonnet 4.6 (not Kimi — free text may contain PII or context requiring nuanced understanding)
- Anonymized student identifier only
- Fallback: if Sonnet unavailable, queue for async evaluation; student sees "Your answer is being reviewed" and score updates later via push notification

## 3. Evaluation Pipeline Architecture

### 3.1 Pipeline Flow

```
Student submits answer
  │
  ▼
Type Router (reads question.type from question metadata)
  │
  ├─► Deterministic Evaluator (MCQ, Numeric, Ordering, Fill-Blank, Diagram)
  │     │
  │     ▼
  │   Direct comparison against answer key
  │   Partial credit computed per type-specific rules
  │
  ├─► CAS Evaluator (Mathematical Expression)
  │     │
  │     ▼
  │   Python sidecar: SymPy service behind FastAPI
  │   1. Parse student expression (sympify with safe_mode)
  │   2. Parse reference expression
  │   3. Symbolic equivalence: simplify(student - ref) == 0
  │   4. Timeout fallback (>2s): numeric sampling at 5 points
  │   5. Partial credit: check intermediate steps if defined
  │
  ├─► LLM Evaluator — Classification (True/False justification)
  │     │
  │     ▼
  │   Kimi K2.5 via LLM ACL (no PII — anonymized payload)
  │   Structured rubric classification: correct / partial / incorrect
  │
  └─► LLM Evaluator — Free-Text (Explanation questions)
        │
        ▼
      Claude Sonnet 4.6 via LLM ACL
      Rubric-based multi-criteria evaluation
      Returns continuous score + written feedback
  │
  ▼
Result Object:
{
  "is_correct": boolean,
  "score": float,              // 0.0–1.0, supports partial credit
  "partial_credit_score": float,
  "error_type": string | null, // "sign_error", "unit_error", "conceptual", "procedural", "careless", null
  "feedback": string,          // human-readable feedback for the student
  "evaluation_method": "deterministic" | "cas" | "llm_classification" | "llm_freetext",
  "evaluation_latency_ms": int,
  "evaluator_model": string | null  // "kimi_k2.5" | "sonnet_4.6" | null (for deterministic)
}
  │
  ▼
Feeds into StudentActor's ConceptAttempted event
  → BKT update (P(known) adjusted based on is_correct)
  → Engagement context (XP award based on score)
  → Stagnation detector (error_type pattern tracking)
  → Spaced repetition (half-life adjusted)
```

### 3.2 Evaluation Latency Targets

| Evaluator | Target p50 | Target p99 | Hard Timeout |
|-----------|-----------|-----------|-------------|
| Deterministic | <5ms | <20ms | 100ms |
| CAS (SymPy) | <200ms | <1s | 2s (then numeric fallback) |
| LLM Classification (Kimi) | <500ms | <2s | 5s |
| LLM Free-Text (Sonnet) | <1s | <3s | 10s |

### 3.3 Fallback Chain

If an evaluator fails or times out:
1. **CAS timeout** → Numeric sampling fallback (5 random points, relative error < 1e-6)
2. **CAS failure** → Queue for async evaluation; return "Answer recorded, result pending"
3. **Kimi unavailable** → If T/F part is correct, award 0.6 (assume justification partial). If T/F wrong, award 0.0
4. **Sonnet unavailable** → Queue for async evaluation; push notification when result is ready
5. **All LLM unavailable** → Deterministic-only mode; LLM-evaluated question types are temporarily skipped in item selection

### 3.4 Error Classification Taxonomy

The `error_type` field in the evaluation result categorizes the student's mistake to feed the MCM graph and stagnation detector:

| Error Type | Description | Example |
|-----------|-------------|---------|
| `conceptual` | Fundamental misunderstanding of the concept | Using addition instead of multiplication for area |
| `procedural` | Correct concept, wrong execution of steps | Correct formula, arithmetic mistake in step 3 |
| `sign_error` | Incorrect sign (positive/negative) | -3 instead of 3 |
| `unit_error` | Wrong or missing unit | Answer in meters when centimeters required |
| `careless` | Answer very close to correct, likely attention lapse | Off by a factor of 10, reversed digits |
| `incomplete` | Partially correct, missing components | Correct derivative but forgot the chain rule factor |
| `terminology` | Wrong technical term used | "Nucleus" instead of "mitochondria" |
| `ordering` | Correct items but wrong sequence | Steps in wrong order |

For deterministic evaluators, error type is inferred from the specific wrong answer (using distractor metadata for MCQ, tolerance analysis for numeric). For LLM evaluators, error type is explicitly classified by the model.

## 4. Hint Escalation System

All question types support a 3-level hint escalation:

| Level | Name | Purpose | XP Penalty | Trigger |
|-------|------|---------|-----------|---------|
| L1 | Nudge | Redirect attention without revealing information | 10% of question XP | Student requests hint OR 1 incorrect attempt |
| L2 | Scaffold | Provide structural support (eliminate options, show steps, give partial info) | 30% of question XP | Student requests after L1 OR 2 incorrect attempts |
| L3 | Near-Answer | Bring student very close to the answer as a teaching moment | 50% of question XP | Student requests after L2 OR 3 incorrect attempts |

**Hint generation:**
- Hints are pre-authored during content authoring (stored in question metadata)
- For free-text and expression types, Sonnet can generate dynamic hints based on the student's specific wrong answer (personalized scaffolding)
- Dynamic hint generation is an LLM call via the ACL, counted toward the per-student cost cap

**After L3 + still incorrect:** Reveal the full answer with a worked explanation. No further XP for this question. The concept is flagged for re-visitation in the next session via spaced repetition.

## 5. Diagnostic Quiz Algorithm (Onboarding)

### 5.1 Purpose

When a new student onboards, the system must efficiently map their existing knowledge across the subject's concept graph (approximately 2,000 nodes for Math). The diagnostic quiz does this in 10–15 questions using an adaptive algorithm based on Knowledge Space Theory (KST).

### 5.2 Knowledge Space Construction

**Definition:** A *knowledge state* is a subset of concepts that a student has mastered. Not all subsets are feasible — prerequisite relationships constrain which combinations are possible.

**Construction from the prerequisite graph:**
1. The domain graph defines prerequisite edges: if concept B requires concept A, then any valid knowledge state containing B must also contain A
2. The set of all prerequisite-consistent subsets forms the *knowledge space* K
3. For a graph with 2,000 nodes, the raw power set is 2^2000, but prerequisite constraints reduce the feasible state space dramatically — typically to O(thousands) of meaningful states for assessment purposes

**Practical state space reduction:**
- Concepts are grouped into **assessment clusters** (10–30 concepts per cluster, grouped by topic and difficulty tier)
- The diagnostic operates at the cluster level, not the individual concept level
- A cluster is "known" if the student answers correctly on a representative item from that cluster
- With approximately 80–120 assessment clusters per subject, the feasible state space is manageable (~10,000–50,000 states after prerequisite pruning)
- Post-diagnostic, individual concept mastery within a cluster is refined during the first few learning sessions

### 5.3 Feasible State Enumeration

At startup (or on curriculum publication), the system pre-computes:
1. The set of assessment clusters and their prerequisite partial order
2. The feasible knowledge states (prerequisite-closed subsets of clusters)
3. For each diagnostic question, the set of states where the student would likely answer correctly

This is a one-time computation per curriculum version, stored alongside the domain graph in the in-memory cache.

### 5.4 Item Selection Algorithm

The diagnostic uses an information-theoretic adaptive item selection strategy inspired by ALEKS:

**Definitions:**
- Let S be the set of feasible knowledge states
- Let P(s) be the current posterior probability of state s (initially uniform over S)
- For a candidate question q targeting cluster c, define:
  - S_correct(q) = { s in S : c in s } (states where the student knows the cluster)
  - S_incorrect(q) = { s in S : c not in s }

**Selection criterion — maximum expected entropy reduction:**

For each candidate question q, compute the expected posterior entropy:

```
H_expected(q) = P(correct|q) * H(posterior|correct) + P(incorrect|q) * H(posterior|incorrect)

where:
  P(correct|q) = sum over s in S_correct(q) of P(s)
  P(incorrect|q) = 1 - P(correct|q)
  H(posterior|correct) = entropy of P(s|correct) for s in S
  H(posterior|incorrect) = entropy of P(s|incorrect) for s in S
```

Select the question q* that minimizes H_expected(q) — i.e., the question whose answer will maximally reduce uncertainty about the student's state.

**Fringe-based heuristic (for computational efficiency):**
- The **inner fringe** of a state s is the set of clusters in s that have at least one prerequisite not in s's successors — i.e., concepts the student probably just barely knows
- The **outer fringe** is the set of clusters not in s that have all prerequisites in s — concepts just beyond the student's current knowledge
- Prioritize questions that probe the aggregate fringe (union of fringes weighted by posterior probability) — these questions are most discriminating

### 5.5 Bayesian Posterior Update

After each student response, update the posterior over all feasible states:

**Likelihood model:**
```
P(correct | student knows cluster c) = 1 - p_slip
P(correct | student doesn't know cluster c) = p_guess
```

**Default parameters:**
| Parameter | Value | Rationale |
|-----------|-------|-----------|
| p_slip | 0.10 | 10% chance of careless error on a known concept |
| p_guess | 0.25 | MCQ with 4 options — random guessing baseline |

**Pre-launch calibration plan** (these defaults are starting points — must be validated empirically):
1. **Data collection**: First 500 diagnostic quiz completions (beta users) generate ~5,000–7,500 response records (10–15 questions each). This is sufficient for MLE parameter estimation.
2. **Calibration method**: Fit p_slip and p_guess per question type via expectation-maximization (EM) on the response matrix, using the `pyBKT` library's built-in parameter fitting (see `docs/intelligence-layer.md` Flywheel 2).
3. **Validation**: Compare fitted parameters to defaults. If fitted p_slip differs from 0.10 by more than ±0.05, or fitted p_guess differs from 0.25 by more than ±0.10, replace defaults with fitted values.
4. **Ongoing recalibration**: Quarterly, as part of the Flywheel 2 BKT retraining cycle. Parameters are versioned (V1, V2, ...) and A/B tested before rollout.
5. **Guard rail**: p_slip is capped at [0.01, 0.30] and p_guess at [0.01, 0.50] to prevent degenerate solutions from noisy data.

For non-MCQ diagnostic questions, p_guess is adjusted:
| Question Type | p_guess |
|--------------|---------|
| MCQ (4 options) | 0.25 |
| True/False | 0.50 |
| Numeric | 0.05 |
| Expression | 0.02 |

**Update rule (Bayes):**
```
P(state s | answer to q) ∝ P(answer | state s, question q) * P(state s)

where:
  If answer is correct:
    P(answer | s, q) = (1 - p_slip)  if cluster(q) in s
    P(answer | s, q) = p_guess       if cluster(q) not in s
  If answer is incorrect:
    P(answer | s, q) = p_slip        if cluster(q) in s
    P(answer | s, q) = (1 - p_guess) if cluster(q) not in s
```

After update, renormalize P(s) so that all probabilities sum to 1.

### 5.6 Stopping Criterion

The diagnostic terminates when **either** condition is met:

1. **Entropy threshold:** The Shannon entropy of the posterior drops below a threshold such that the most probable state has posterior probability > 0.90 (student's state is determined with >90% confidence)

   ```
   Stop if: max(P(s)) > 0.90
   ```

2. **Hard cap:** 15 questions reached, regardless of entropy. Use the current posterior as-is. In practice, 10–12 questions typically suffice for >90% confidence.

3. **Early termination:** If after 5 questions, a single state has posterior > 0.95, stop early (the student's level is very clear — e.g., complete beginner or very advanced).

### 5.7 Initial Knowledge Map Generation

From the final posterior distribution, compute the marginal probability for each assessment cluster:

```
P(cluster c is known) = sum over { s in S : c in s } of P(s)
```

Then propagate to individual concepts within each cluster and assign mastery status:

| P(known) | Status | UI Color | Meaning |
|----------|--------|----------|---------|
| >= 0.85 | Mastered | Green | Student likely knows this — skip in learning path, include in spaced repetition (matches mastery threshold in `docs/event-schemas.md` ConceptMastered_V1) |
| 0.3 – 0.85 | In-Progress | Yellow | Uncertain — prioritize for early learning sessions to refine estimate |
| < 0.3 | Unknown | Gray | Student likely doesn't know this — will be taught when prerequisites are met |

**Initial BKT parameters:** The diagnostic result initializes the BKT model's P(known) prior for each concept. This replaces the default uniform prior (0.5) with an informed estimate, accelerating the adaptive learning from the first session.

### 5.8 Edge Cases

| Scenario | Handling |
|----------|---------|
| **Student skips a question** | Treated as P(correct) = p_guess (assume unknown). The question's cluster leans toward "not known" in the posterior update, but not as strongly as an explicit wrong answer |
| **Suspiciously fast answer** (<3 seconds on a non-trivial question) | Flag as potential guess. Reduce the update weight: use p_effective_slip = 0.3, p_effective_guess = 0.5 (both closer to 0.5, making the response less informative). Log for analytics |
| **Network interruption** | Diagnostic state (current posterior, question history) is persisted after each answer in the StudentActor's event store. On reconnect, resume from the last answered question. No data loss |
| **Student exits and returns later** | Same as network interruption — state is durable. Diagnostic can be resumed within 7 days. After 7 days, restart diagnostic (student's knowledge may have changed) |
| **Student is exactly at the boundary** (posterior never converges) | After 15 questions (hard cap), use current posterior. The "in-progress" (yellow) classification handles uncertainty — these concepts will be clarified in the first few learning sessions |
| **Empty knowledge** (student answers everything wrong) | Valid state — all clusters mapped to "unknown" (gray). Learning path starts from foundational concepts. This is expected for students who are just beginning the subject |
| **Full knowledge** (student answers everything correct) | Valid state — all tested clusters mapped to "mastered" (green). System assigns advanced concepts and challenges. Spaced repetition schedules are initialized immediately |

### 5.9 Diagnostic Question Bank

Diagnostic questions are **separate from the regular question bank**:

| Attribute | Diagnostic Questions | Regular Questions |
|-----------|---------------------|-------------------|
| **Purpose** | Maximum state-space partitioning | Teaching and practice |
| **Design criterion** | High discrimination — each question cleanly separates "knows" from "doesn't know" | Varied Bloom's levels, hint support, misconception targeting |
| **Calibration** | Pre-calibrated with known discrimination parameters (item difficulty, slip/guess rates) | Parameters learned over time from student interactions |
| **Count per cluster** | 2–3 questions (to allow adaptive selection without repetition) | 8–15 questions per concept |
| **Hint support** | No hints — pure assessment | Full 3-level hint escalation |
| **XP awarded** | No XP — diagnostic is not gamified | Full XP per question |
| **Reuse** | Retired after diagnostic; never shown again in learning sessions | Reusable with spaced repetition scheduling |
| **Evaluation** | Deterministic only (MCQ, numeric) — no LLM dependency during onboarding | All 8 types including LLM-evaluated |

**Rationale for deterministic-only diagnostic:** The onboarding flow must be fast and reliable. LLM latency and potential unavailability during the critical first-impression onboarding experience is an unacceptable risk. MCQ and numeric questions provide sufficient discrimination for cluster-level assessment.

## 6. Integration Points

### 6.1 With StudentActor (Learner Context)

- Diagnostic result initializes the `StudentActor`'s knowledge overlay (P(known) per concept)
- Each question attempt during regular sessions produces a `ConceptAttempted` event with the evaluation result
- The `error_type` field feeds the MCM graph for methodology selection
- BKT updates use `is_correct` and `score` from the evaluation result

### 6.2 With Pedagogy Context

- Item selection reads question metadata (type, Bloom's level, difficulty) to choose the next question
- The `LearningSessionActor` routes each answer to the appropriate evaluator via the Type Router
- Hint requests are managed by the session actor, tracking escalation level per question

### 6.3 With Content Authoring Context

- Question metadata (type, evaluation method, answer key, rubric, hints, diagnostic flag) is authored and published through the content pipeline described in `docs/content-authoring.md`
- QA checks enforce minimum question counts per type and Bloom's level coverage

### 6.4 With Delivery Context (LLM ACL)

- LLM-evaluated question types (T/F justification, free-text explanation) route through the ACL
- The ACL applies the privacy rules: Kimi receives only anonymized structured data; Sonnet receives anonymized free text
- CAS evaluation (SymPy) runs as a sidecar behind the same FastAPI layer, sharing the gRPC transport
