> **Audience:** Product team, engineering team, design team

## 1. Parent Experience (I6)

### 1.1 Context

Parents are the purchase decision-makers. They currently pay ~3,000 NIS/subject for private Bagrut tutoring (5 STEM subjects = 15,000 NIS/year). Cena costs 799 NIS/year for all subjects — a 94% saving. The parent dashboard must make this value visible and continuous so parents feel confident in their subscription.

Parents need to see **progress, consistency, and readiness** without invading their teenager's learning space.

### 1.2 Parent Dashboard (mobile + web)

#### 1.2.1 Overview Screen

The landing screen after login. Designed for a 30-second glance.

| Element | Description |
|---------|-------------|
| **Child's profile header** | Name, grade (e.g., "11th grade"), subjects enrolled (e.g., Math 5-unit, Physics 5-unit, Chemistry 5-unit) |
| **Streak count + streak calendar** | Current streak count (e.g., "12-day streak") with a visual calendar heatmap (GitHub contribution graph style) showing daily activity intensity over the past 12 weeks. Color intensity = study time that day |
| **Weekly study summary** | This week: X sessions, Y minutes total, Z concepts mastered. Displayed as three cards with icons |
| **Bagrut readiness scores** | Per-subject percentage with trend arrow (up/down/stable). Example: "Math 5-unit: 72% ready (up 4% this week)". Tapping a subject navigates to the subject detail screen |

**Data sources:** Analytics Context CQRS projections from `ConceptMastered`, `SessionCompleted`, and Engagement Context streak events. Served via GraphQL.

#### 1.2.2 Subject Detail Screen

Accessed by tapping a subject from the overview screen. Provides deeper insight into a single subject.

| Element | Description |
|---------|-------------|
| **Simplified knowledge graph** | Cluster-level view (not individual concept nodes). Shows major topic clusters (e.g., "Calculus", "Trigonometry", "Probability") as colored circles. Color = mastery level (green = mastered, yellow = in progress, gray = not started, red = struggling). Clusters are connected by prerequisite arrows. This is a simplified projection of the full student knowledge graph — parents see ~15-25 clusters per subject, not the 500-2,000 individual concept nodes |
| **Mastery percentage** | "X% of syllabus mastered" — computed as the ratio of mastered concept nodes to total concept nodes in the subject's knowledge graph |
| **Estimated completion time** | "At current pace, estimated to complete this subject by [date]". Calculated from the student's average concept mastery rate over the past 30 days, extrapolated to remaining unmastered concepts. Displayed only when sufficient data exists (minimum 14 days of activity) |
| **Recent activity** | Last 5 concepts worked on, each showing: concept name, status (mastered / in-progress / struggling), and date last attempted |
| **Cohort comparison** | "Your child is ahead of / behind / on-track with 72% of students in this subject." Anonymized, opt-in only (both the viewing parent and contributing students must opt in). Based on percentile rank within the same grade and subject. Displayed only when cohort size >= 30 students to ensure meaningful anonymization |

**Data sources:** Learner Context knowledge graph overlay (projected to cluster-level aggregation), Pedagogy Context session history, Analytics Context cohort percentile projections.

#### 1.2.3 Weekly Digest (push notification + email)

Delivered every **Sunday evening** (before the Israeli school week begins). Available as push notification summary + full email, with optional WhatsApp delivery.

| Element | Description |
|---------|-------------|
| **Study time comparison** | "This week: 3h 20m (up from 2h 45m last week)" |
| **New concepts mastered** | "Mastered 8 new concepts this week: [list top 3 by subject]" |
| **Subjects needing attention** | "Physics has had no activity in 5 days — consider encouraging a session" (shown only for subjects with >3 days of inactivity) |
| **Streak status** | "Current streak: 12 days" or "Streak broken on Wednesday — your child can start a new one today" |
| **Upcoming Bagrut exam dates** | "Math Bagrut: January 15 — your child is 72% ready (on track)" with readiness estimate per subject. Exam dates sourced from Ministry of Education published schedule |

**Delivery:** Outreach Context routes the digest to the parent's preferred channel(s). The digest is generated by the Analytics Context as a scheduled CQRS projection, published to NATS, and picked up by the Outreach Context for multi-channel delivery.

#### 1.2.4 Settings Screen

| Setting | Description |
|---------|-------------|
| **Notification preferences** | Toggle channels: weekly email digest (default: on), push notifications (default: on), WhatsApp (default: off). Per-notification-type granularity: digest, milestones, risk alerts |
| **Study goal setting** | Parent sets a target, e.g., "I want my child to be 80% ready for Math Bagrut by January 15." The system tracks progress toward this goal and includes it in the weekly digest. Goals are visible to the student (as motivation, not pressure — framed as "your family's goal") |
| **Privacy controls** | Toggle what the parent can see vs. what stays private to the student. Options: cohort comparison (opt-in/opt-out), study time visibility (on by default), streak visibility (on by default). Student annotations and session-level detail are always private (not toggleable — this is a system-level privacy boundary, not a preference) |

### 1.3 Privacy Boundary

The parent-student privacy boundary is a core architectural constraint, not a configurable preference.

| Data Category | Parent Can See | Parent Cannot See |
|---------------|---------------|-------------------|
| Aggregate progress (mastery %, concepts mastered count) | Yes | — |
| Streak count and calendar | Yes | — |
| Study time (total per day/week) | Yes | — |
| Bagrut readiness score | Yes | — |
| Cluster-level knowledge graph | Yes | — |
| Individual concept status (mastered/in-progress/struggling) | Yes (last 5 in subject detail) | — |
| Cohort comparison percentile | Yes (if opted in) | — |
| Individual question text and answers | — | Yes |
| Student annotations and notes | — | Yes |
| Session-level detail (exact questions, time per question, number of attempts per question) | — | Yes |
| Methodology being used per concept | — | Yes |
| Notification history (what outreach the student received) | — | Yes |
| Stagnation signals and methodology switch events | — | Yes |

**Rationale:** This boundary respects teenage autonomy while giving parents the confidence signal they need to justify the subscription. Parents see *that* their child is learning and *how much*, but not *exactly what mistakes they made or what they wrote*. This prevents helicopter parenting dynamics that would cause the student to disengage from the platform.

**Implementation:** The Analytics Context maintains separate CQRS read models for parent views. These projections aggregate from the event store but exclude private event types (`AnnotationAdded`, `ExerciseAttempted` detail, `MethodologySwitched`). The GraphQL schema enforces this boundary at the resolver level — parent-authenticated queries cannot access student-private projections.

### 1.4 Parent Notifications

| Notification Type | Trigger | Channel | Timing |
|-------------------|---------|---------|--------|
| **Weekly digest** | Scheduled | Push + email (+ WhatsApp if opted in) | Sunday evening, 19:00 Israel time |
| **Milestone celebration** | `ConceptMastered` event crosses a threshold (25%, 50%, 75%, 100% of a subject) | Push notification (immediate) | Real-time on threshold crossing |
| **Risk alert** | Study time decreased >40% week-over-week, OR >7 days of inactivity in any enrolled subject | Push notification | Monday morning, 08:00 Israel time (only if opt-in enabled for risk alerts) |
| **Exam countdown** | 30 days, 14 days, and 7 days before a Bagrut exam | Push + email | Morning of the countdown date |

**Risk alert opt-in:** Risk alerts are off by default. The parent must explicitly enable them in Settings. The alert text is carefully worded to encourage supportive conversation, not punitive action. Example: "Study time has decreased this week. A quick check-in might help — sometimes a small conversation goes a long way."

## 2. Teacher Dashboard (I10)

### 2.1 Context

The B2B2C model depends on school partnerships. Teachers are the gatekeepers — if Cena makes their job easier, they become advocates. The teacher dashboard must be **indispensable**: it should surface insights that a teacher cannot get from any other source, specifically class-level knowledge gap identification and per-student risk detection.

The teacher dashboard is **web-only at MVP**. Mobile is deferred until post-launch validation confirms teacher demand for mobile access.

### 2.2 Teacher Dashboard Screens

#### 2.2.1 Class Overview

The default landing screen for teachers. Shows all students in a selected class.

| Element | Description |
|---------|-------------|
| **Student list** | Table with columns: student name, mastery % per subject, current streak, last active date. Sortable by any column, filterable by subject |
| **Traffic light indicators** | Per student, per subject: **Green** (on-track: mastery % is at or above the expected pace for Bagrut readiness by exam date), **Yellow** (falling behind: mastery % is 10-25% below expected pace), **Red** (at risk: mastery % is >25% below expected pace OR >14 days inactive). Thresholds are configurable per class by the teacher |
| **Class average vs. national average** | When sufficient data permits (minimum 100 students nationally in the same subject/grade), display class average mastery % alongside national average. This becomes available as Cena scales beyond the initial school partners |
| **Quick actions** | Per student: "View detail", "Assign review". Per class: "View knowledge gaps", "Create assignment", "Export report" |

**Data sources:** School Context class membership, Analytics Context CQRS projections (student mastery percentages, streak data, last activity timestamps).

#### 2.2.2 Class Knowledge Gap Heatmap

The highest-value screen for teachers. Enables data-driven lesson planning.

| Element | Description |
|---------|-------------|
| **Matrix layout** | Rows = concepts in the selected unit/topic (teacher selects from the curriculum knowledge graph). Columns = students in the class. Cell color indicates mastery status |
| **Cell colors** | **Green** = mastered. **Yellow** = in progress (attempted but not yet mastered). **Red** = struggling (multiple attempts, accuracy below threshold). **Gray** = unknown (not yet attempted) |
| **Row aggregation** | Each concept row shows a summary: "X/Y students have mastered this concept" and a percentage bar. Rows are sortable by mastery percentage (ascending = most problematic concepts first) |
| **Column aggregation** | Each student column shows overall mastery % for the selected unit |
| **Interaction** | Clicking a concept (row header) expands to show which specific students are struggling, in progress, or haven't attempted it. Clicking a student (column header) navigates to the student detail view |

**Key insight this enables:** "80% of my class struggles with integration by parts — I need to reteach this in class tomorrow." This is the kind of actionable intelligence that makes Cena indispensable for teachers.

**Data sources:** Learner Context knowledge graph overlays for all students in the class, filtered by the teacher's selected unit. Projected as a dedicated CQRS read model in the Analytics Context, scoped to the School Context class membership.

#### 2.2.3 Student Detail (Teacher View)

Accessed from the class overview or heatmap. Provides more granularity than the parent view but respects the same privacy boundaries for student-private data.

| Element | Description |
|---------|-------------|
| **Overview** | Same information as the parent overview screen: mastery %, streak, study summary, Bagrut readiness |
| **Knowledge graph** | Full cluster-level view (same as parent view — clusters, not individual nodes) per subject |
| **Active methodology indicator** | Shows the current primary methodology being used for this student (e.g., "Socratic dialogue", "spaced repetition", "worked examples"). The teacher can see *what* methodology is active but not the internal scoring or switching logic |
| **Stagnation signals** | If the student has active stagnation flags, the teacher sees a warning: "This student may be stagnating on [concept cluster]. The system is adapting its approach." No detail on the specific signals or methodology switch mechanics |
| **Manual methodology suggestion** | Teacher can click "Suggest methodology change" — this creates a signal that the `MethodologySwitchService` considers in its next evaluation. The teacher cannot force a switch; the system decides. This preserves the adaptive engine's integrity while giving teachers a voice |
| **Assign concepts** | Teacher can select specific concepts and assign them for review. These appear as a "teacher quest" in the student's app (see Assignment Feature below) |

**What the teacher CANNOT see** (same privacy boundary as parents, plus):
- Student annotations and notes
- Individual question text and answers
- Exact session times and duration per session
- Notification history (what outreach the student received)
- Methodology switch history and decision details

#### 2.2.4 Assignment Feature

Teachers can create targeted assignments using the curriculum knowledge graph.

| Element | Description |
|---------|-------------|
| **Concept selection** | Teacher browses the curriculum knowledge graph (organized by unit/topic) and selects one or more concepts. Can select individual concepts or entire topic clusters |
| **Assignment target** | Assign to: entire class, a subset of students (selected from class list), or individual students. The heatmap "struggling students" list can be directly converted to an assignment target |
| **Student experience** | Students see teacher assignments as a "Teacher Quest" in their app — a distinct card in their learning feed, branded with the teacher's name and a quest icon. The quest follows Cena's adaptive methodology (the teacher selects *what* to practice, not *how*) |
| **Completion tracking** | Teacher sees a completion dashboard: per student, has the quest been started / in progress / completed. For completed quests, the teacher sees aggregate mastery outcome (mastered / needs more practice) but not individual question-level results |
| **Due dates** | Optional due date. Students receive a reminder notification 24 hours before the due date via their preferred channel |

**Data flow:** Teacher creates assignment -> School Context persists assignment -> NATS event `TeacherQuestAssigned` -> Student actors receive the quest -> Pedagogy Context generates adaptive exercises for the assigned concepts -> completion events flow back through Analytics Context to the teacher's completion dashboard.

#### 2.2.5 Exam Preparation View

Designed for the weeks leading up to Bagrut exams.

| Element | Description |
|---------|-------------|
| **Exam configuration** | Teacher selects: upcoming exam date, subject, and topics/units covered on the exam (selected from the curriculum knowledge graph) |
| **Class readiness dashboard** | Per topic: percentage of class that has mastered the topic, percentage in progress, percentage not yet attempted. Displayed as horizontal stacked bar charts |
| **At-risk students** | List of students most at risk of failing, ranked by gap between current mastery and exam readiness threshold (configurable, default 70%). Each student shows: current mastery %, gap to threshold, days remaining, estimated study time needed |
| **Focus area recommendations** | Based on aggregate class knowledge gaps: "Recommend focusing classroom instruction on: [Topic A] (only 35% mastery), [Topic B] (42% mastery)." These are the topics where the most students are struggling, making classroom time most impactful |
| **Quick assign** | One-click to create a teacher quest from any focus area recommendation, targeting the students who need it most |

#### 2.2.6 Reports (Exportable PDF)

Three standard report types, all exportable as branded PDF documents.

| Report | Content | Use Case |
|--------|---------|----------|
| **Class progress report** | Per subject, per month: class mastery % trajectory over time, concepts mastered by the class, average study time, active student count, knowledge gap highlights | Monthly school administration reporting, department meetings |
| **Individual student report** | Student mastery % per subject, streak history, Bagrut readiness per subject, strengths (top mastered clusters), areas for improvement (struggling clusters), study consistency metrics | Parent-teacher conferences. Provides a structured, data-driven conversation framework |
| **Concept mastery distribution report** | Per concept in a selected unit: histogram showing the distribution of student mastery levels across the class. Identifies bimodal distributions (some students get it, some don't) | Curriculum planning, identifying concepts that need pedagogical redesign at the school level |

**Report generation:** Analytics Context generates reports on demand from CQRS read models. PDF rendering via a server-side template engine (e.g., Puppeteer or wkhtmltopdf). Reports include the school logo (uploaded during school onboarding) and Cena branding.

### 2.3 Teacher Onboarding Flow

| Step | Action | Detail |
|------|--------|--------|
| 1 | **School admin creates account** | School admin registers the school, uploads school logo, configures school-wide settings (e.g., which subjects are offered). School admin receives a school code |
| 2 | **School admin adds teachers** | Admin invites teachers via email. Teacher receives an invitation link, creates their account, and is associated with the school |
| 3 | **Teacher creates class** | Teacher creates a class (e.g., "11th Grade Math 5-unit, Section A"), which generates a unique class code |
| 4 | **Students join via class code** | Students enter the class code in their existing Cena app. This links their Cena account to the teacher's class. Students who don't yet have a Cena account are prompted to create one |
| 5 | **Data visibility starts from join date** | The teacher can only see student data from the date the student joined the class. Historical data (from before the student linked to the class) is not retroactively shared. This is a privacy boundary — students opt in to sharing with a specific teacher by joining their class |

**Multi-class support:** Students can join multiple classes (e.g., Math teacher + Physics teacher). Each teacher sees only their own subject's data for that student.

### 2.4 Data Access Rules

| Role | Can See | Cannot See |
|------|---------|------------|
| **Teacher** | Their own classes only. Per student: mastery levels, streak, last active date, active methodology (name only), stagnation warnings, assignment completion. Aggregate: class knowledge gap heatmap, class averages, cohort comparisons | Other teachers' classes. Student annotations. Individual question answers. Exact session timestamps. Notification history. Methodology switch decision details. Data from before a student joined the class |
| **School admin** | All classes in the school. Same per-student and aggregate visibility as teachers, across all classes. School-wide reports and analytics | Student annotations. Individual question answers. Exact session timestamps. Notification history. Methodology switch decision details. Data from before a student joined any class in the school |
| **Cena system** | All data (required for adaptive engine operation) | N/A — the system is the source of truth |

**Enforcement:** Data access rules are enforced at multiple layers:
1. **GraphQL resolvers** — queries are scoped by the authenticated user's role and class membership
2. **CQRS read models** — separate projections for teacher views exclude private event types
3. **School Context** — wraps student data with school-scoped and class-scoped access filters
4. **Audit log** — all teacher data access is logged for compliance (which teacher viewed which student's data, when)

## 3. Interaction Between Stakeholder Views

### 3.1 Data Flow Summary

```
Student Activity
    │
    ▼
StudentActor (event-sourced)
    │
    ├──► Domain Events (NATS JetStream)
    │       │
    │       ├──► Analytics Context
    │       │       │
    │       │       ├──► Parent read model (aggregate, privacy-filtered)
    │       │       │       └──► Parent Dashboard (GraphQL)
    │       │       │       └──► Weekly Digest (Outreach Context)
    │       │       │
    │       │       ├──► Teacher read model (class-scoped, privacy-filtered)
    │       │       │       └──► Teacher Dashboard (GraphQL)
    │       │       │       └──► Reports (PDF generation)
    │       │       │
    │       │       └──► Internal analytics (full detail, ops only)
    │       │
    │       └──► School Context (class membership filtering)
    │
    └──► Student's own view (full detail, no privacy filtering)
```

### 3.2 Shared Privacy Principles

1. **Students own their data.** Annotations, detailed session logs, and methodology details are private to the student and the Cena adaptive engine.
2. **Parents see outcomes, not process.** Mastery levels, study time, and readiness — not individual mistakes or personal notes.
3. **Teachers see what helps them teach.** Knowledge gaps, at-risk students, and aggregate performance — not private learning details.
4. **Data sharing is opt-in and scoped.** Students opt in by joining a class (teacher access). Parents see data by virtue of the subscription relationship, but cohort comparisons require explicit opt-in.
5. **Historical data is not retroactive.** Teachers see only post-join data. This prevents surveillance concerns when a student switches classes or schools.
