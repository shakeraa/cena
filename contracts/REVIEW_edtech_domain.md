## EDTECH DOMAIN REVIEW — Dr. Noa Shapira (Adaptive Learning Architect, ex-Squirrel AI / ALEKS)

**Date:** 2026-03-26
**Scope:** All contracts in `llm/`, `mobile/lib/features/`, `data/neo4j-schema.cypher`, `backend/domain-services.cs`
**Lens:** Pedagogical soundness, learning science alignment, gamification ethics, adaptive model validity

---

### CRITICAL (will harm learning outcomes)

1. **Mastery threshold 0.85 is too low for gating progression** — `domain-services.cs:121` `MasteryThreshold = 0.85` — The classic BKT literature (Corbett & Anderson 1994) recommends 0.95 as the mastery gate. At 0.85, ~15% of students who "master" a concept actually have not learned it (type II error). This is especially dangerous in a prerequisite-gated DAG: a student declared "mastered" at Addition with P(known)=0.86 will be routed to Multiplication with a shaky foundation. The justification may be "reduce frustration," but the research (Doignon & Falmagne 1999, Knowledge Spaces) shows that premature advancement is the #1 cause of cascading failure in prerequisite chains. **Fix:** Use 0.95 for hard-gated prerequisites (strength=1.0 in Neo4j) and 0.85 for soft/recommended prerequisites (strength<1.0). This two-tier threshold respects both rigor and student experience.

2. **BKT has no P(forget) in the default path — yet spaced repetition exists** — `domain-services.cs:117` `PForget` is defined but the comment says "Often 0 in classic BKT." If P(forget)=0, the BKT model is monotonically non-decreasing: once you reach mastery, you can never un-master a concept through BKT alone. This means the HLR spaced repetition system (Flywheel 4) and the BKT model (Flywheel 2) are **incoherent** — HLR assumes recall decays, but BKT says mastery never decays. When a student returns after 2 weeks with low recall, BKT still says P(known)=0.92 while HLR says P(recall)=0.40. Which does the system trust? There is no reconciliation contract defined. **Fix:** Either (a) implement P(forget) > 0 (e.g., 0.01-0.05) so BKT can degrade, and use BKT as the single mastery source; or (b) define an explicit "mastery override" rule: if HLR predicts recall < 0.5 AND time since last practice > 7 days, reset BKT mastery to max(0.3, HLR_recall). This must be specified in the contract.

3. **Stagnation detection will fire false positives for slow learners** — `domain-services.cs:457-460` — The stagnation trigger is "score > 0.7 for 3 consecutive sessions." But the accuracy plateau normalization formula (`sigmoid(10 * (0.05 - improvement_rate))`) triggers stagnation when improvement is less than 5% between 10-question windows. For a student making genuine but slow progress (2-3% improvement per window), this will ALWAYS signal stagnation. The system will then switch methodologies every ~3 sessions, creating the exact "methodology churn" the switching prompt warns against. Slow learners are not stagnating; they are learning at their pace. **Fix:** The improvement threshold should be personalized. Use the student's historical learning rate (from BKT's P(learn) parameter) as the baseline. If the student's P(learn) for this concept is 0.03, then 2% improvement is consistent with their trajectory — not stagnation. Only trigger stagnation when improvement_rate < 0.3 * P(learn) for 3+ sessions.

4. **Error classification routed to Kimi (cheapest model) — but it's the linchpin of adaptive routing** — `acl-interfaces.py:297-301` — Error classification is routed to Kimi K2 0905 to save cost, but its output directly drives the MCM methodology switch: `(ErrorType, ConceptCategory) -> Methodology`. A misclassification of "conceptual" as "procedural" means the system prescribes worked_examples when it should prescribe socratic — the student drills execution of a procedure they fundamentally don't understand. In the Squirrel AI literature, error classification accuracy below 85% degrades the entire adaptive loop. Hebrew math answers add complexity (RTL notation, Hebrew-specific phrasing). **Fix:** Route error classification to Sonnet for the first 50 attempts per student (until a local pattern is established), then downgrade to Kimi only for students whose Kimi-vs-Sonnet agreement rate exceeds 90% (measured via shadow-scoring A/B).

5. **No prerequisite mastery validation before concept presentation** — The Neo4j schema defines `PREREQUISITE_OF` edges with `strength` properties, and the utility query shows how to traverse prerequisite chains. But there is NO domain service interface that checks prerequisites before starting practice on a concept. The `SessionScreen` in Flutter takes an optional `Subject` but no concept. The `StudentContext` has a `mastery_map` but nothing enforces "do not present Fractions questions if Multiplication mastery < 0.95." The knowledge graph is decorative unless the backend enforces prerequisite gates. **Fix:** Add an `IPrerequisiteGateService` to `domain-services.cs` with a `CanAttempt(studentId, conceptId)` method that checks all hard-gated prerequisites (strength >= 0.8) are mastered. The session question selection loop must call this before presenting any question.

### HIGH (will reduce engagement or retention)

1. **Streak warning creates anxiety, not motivation** — `streak_widget.dart:237-271` `StreakWarning` — The streak warning is a textbook example of loss aversion manipulation (Hamari et al. 2014, "Does Gamification Work?"). A dimming flame icon + countdown timer + "Your streak is about to expire!" is designed to induce anxiety that drives engagement through fear of loss rather than intrinsic motivation. For 12-18 year old students already anxious about Bagrut exams, this compounds stress. Research (Hanus & Fox 2015) shows that loss-framed gamification reduces intrinsic motivation and long-term engagement, even when short-term logins increase. **Fix:** Replace the loss-framed warning with a gain-framed nudge: "You've studied 5 days in a row! Want to keep the momentum going?" No countdown, no dimming flame, no urgency language. The streak should be a record of accomplishment, not a threat to protect.

2. **XP formula rewards volume over mastery** — `streak_widget.dart:80-84` `xpForLevel(n) = 100 * n * (1 + 0.1 * n)` — This is a standard exponential XP curve that rewards answering more questions. But there is no evidence in the contracts that XP is weighted by concept difficulty, mastery improvement, or depth of understanding. A student who answers 50 easy recall questions gets more XP than one who struggles through 10 analysis-level problems and achieves genuine mastery. This creates a perverse incentive: grind easy questions for XP rather than tackle challenging material. **Fix:** XP should be awarded based on mastery delta, not question count. Proposed formula: `XP = base_xp * (1 + bloom_level_multiplier) * mastery_delta_multiplier`. A student who moves from 0.4 to 0.85 mastery on an analysis-level concept should earn 5-10x the XP of answering a recall question they already know.

3. **Challenge cards (SmartyMe-style) have fixed XP rewards regardless of difficulty** — `diagram_models.dart:243` `@Default(10) int xpReward` — All challenge cards award 10 XP by default. Combined with the XP formula above, this means expert-tier challenge cards (analysis/synthesis, red glow) give the same reward as beginner-tier cards (recall, green glow). This is pedagogically backwards — it should feel meaningfully different to tackle harder content. **Fix:** Scale xpReward by ChallengeTier: beginner=5, intermediate=10, advanced=20, expert=40. Also add a "first clear" bonus (2x) to incentivize attempting new challenges over re-grinding known ones.

4. **Feynman mastery threshold (0.75) is disconnected from BKT mastery threshold (0.85)** — `acl-interfaces.py:549` `demonstrates_mastery: True if overall_score >= 0.75` vs `domain-services.cs:121` `MasteryThreshold = 0.85` — A student can "demonstrate mastery" via Feynman at 0.75 but not be considered mastered by BKT until 0.85. This creates confusing mixed signals: the UI says "you've demonstrated mastery!" but the knowledge graph node is still yellow. Conversely, BKT can declare mastery at 0.85 without the student ever demonstrating explanation ability. **Fix:** Feynman should not independently declare "mastery." Instead, a strong Feynman score (>=0.75) should boost the BKT update — treat it as a high-confidence correct answer with a multiplied P(learn) effect. The single mastery source remains BKT.

5. **Cognitive load service measures fatigue proxies, not actual cognitive load** — `domain-services.cs:388-407` `FatigueInput` — The inputs are: session duration, questions attempted, accuracy, response time, hints, skips, and previous fatigue score. None of these directly measure cognitive load. Response time increase could mean the student is thinking more deeply (desirable productive struggle) or is fatigued (undesirable). Accuracy decrease mid-session could be because the adaptive system is increasing difficulty (working as intended) or because the student is exhausted. The model conflates productive struggle with fatigue. **Fix:** Add a "difficulty trajectory" input — if the system is actively increasing difficulty (because the student is succeeding), then slower response times and lower accuracy are EXPECTED and should NOT increase the fatigue score. The fatigue signal should be: `observed_performance - expected_performance_at_current_difficulty`. Also consider asking the student directly with a quick emoji-scale ("How are you feeling?") every 10 questions — self-report is still the best predictor of cognitive load (Paas et al. 2003).

6. **MCM graph has only 3 categories and 3 error types populated at launch — coverage is ~12% of the matrix** — `neo4j-schema.cypher` — The MCM has 3 error types (procedural, conceptual, motivational) x 3 concept categories (arithmetic, algebra, geometry) = 9 cells. Only 8 RECOMMENDS edges are defined, covering arithmetic fully and algebra partially. Geometry has ZERO recommendations. When a student makes a conceptual error on a geometry problem, the MCM lookup returns empty, falling through to "error-type-only defaults" which are not defined in any contract. At launch, the majority of adaptive decisions will hit this fallback path. **Fix:** Define explicit fallback defaults in `domain-services.cs` `IMethodologySwitchService` for when MCM returns empty. Also, pre-populate ALL 9 cells of the initial matrix, even with low confidence (0.5) and small sample_size, based on pedagogical theory rather than data. Worked examples for procedural errors is well-established across ALL categories. These theoretical priors can be overridden as data accumulates.

7. **Leaderboard ranks by streak (effort proxy) but streak is a gamification artifact, not an effort metric** — `streak_widget.dart:287` — The leaderboard shows "study streak" as the primary ranking metric. But a streak only counts days with ANY activity, not quality of effort. A student who opens the app, answers 1 easy question, and closes maintains their streak. A student who does 2 hours of deep practice 4 days a week but skips weekends has no streak. The leaderboard inadvertently punishes deep learners and rewards shallow engagement. **Fix:** Rank by "weekly learning hours" or "mastery concepts improved this week" — metrics that actually correlate with learning. Or better yet, eliminate competitive ranking entirely and show only personal progress ("You improved 3 concepts this week, your best week yet!").

### MEDIUM (missed opportunity for better pedagogy)

1. **Spaced repetition uses HLR but the recall threshold is 0.85 — too high for STEM concepts** — `domain-services.cs:160` `recallThreshold = 0.85` — HLR was developed at Duolingo for vocabulary learning where 0.85 recall makes sense (you either remember the word or you don't). STEM concepts are different: partial recall is meaningful and productive. A student who can't immediately recall the quadratic formula but remembers "something with b-squared" is in a productive retrieval state. Scheduling review when recall drops to 0.85 means interrupting the student for review while they still largely remember the material. **Fix:** Use 0.70 as the recall threshold for STEM concepts and 0.85 for pure memorization items (formulas, definitions). This reduces review frequency by ~40% while still catching genuine forgetting. Literature reference: Bjork & Bjork (2011), "Making Things Hard on Yourself, But in a Good Way."

2. **No interleaving / desirable difficulty in the session model** — The `MethodologyId` enum includes `SPACED_REPETITION` and `GAMIFIED_DRILL` but there is no `INTERLEAVED_PRACTICE` methodology. The `MethodologyIndicator` in `session_screen.dart` lists "interleaved" as a Methodology enum value, but this doesn't appear in `MethodologyId` in `acl-interfaces.py` or the MCM graph. Interleaving (mixing problem types within a session) is one of the strongest findings in learning science (Rohrer 2012, "Interleaving Helps Students Distinguish among Similar Concepts"). The backend has no mechanism to mix concepts within a single session. **Fix:** Add `INTERLEAVED` to `MethodologyId` and define it in the MCM graph. When active, the session should mix 2-3 related concepts (e.g., fractions + negative numbers) rather than blocking on one concept. This is especially effective for near-mastery consolidation.

3. **Hebrew math glossary is comprehensive but static — no student-level customization** — `prompt-templates.py:26-80` — The glossary injects the same terminology for a grade 7 student and a grade 12 student. Grade 7 students don't need "chain rule" or "determinant" terms; injecting them wastes context window tokens and may confuse the LLM. Grade 12 students may need domain-specific Bagrut terminology not in the generic glossary. **Fix:** Partition the glossary by grade band (7-9, 10-12) and concept cluster. Only inject relevant terms for the current concept and grade. This also improves prompt caching efficiency (fewer unique system prompts).

4. **Feedback overlay auto-dismisses after 4 seconds — insufficient for wrong answers** — `session_screen.dart:317` `displayDuration = const Duration(seconds: 4)` — When a student gets an answer wrong, the feedback overlay shows the error explanation for only 4 seconds before auto-dismissing. For conceptual errors with Hebrew mathematical feedback, 4 seconds is barely enough to read the text, let alone process it. Research on feedback timing (Shute 2008, "Focus on Formative Feedback") shows that elaborated feedback needs at least 8-12 seconds of reading time for processing. **Fix:** Scale auto-dismiss duration by error type: `none` (correct) = 2s, `notation` = 4s, `procedural` = 6s, `conceptual` = 10s, `motivational` = 4s (with encouragement). Also, add a "Got it" button and never auto-dismiss for conceptual errors — let the student control when to move on.

5. **Socratic prompt has no metacognitive scaffolding** — `prompt-templates.py:165-245` — The Socratic system prompt instructs the LLM to ask questions that target gaps and misconceptions, but never instructs it to ask metacognitive questions ("How confident are you in that answer?" / "What strategy are you using?" / "Can you think of a way to check your answer?"). Metacognitive questioning is the single highest-effect-size intervention in tutoring research (d=0.69, Hattie 2009). **Fix:** Add a metacognitive question type to the `question_type` enum: `"guiding" | "probing" | "clarifying" | "challenge" | "metacognitive"`. Instruct the LLM to ask one metacognitive question every 3-4 turns, especially when the student answers correctly (to prevent overconfidence).

6. **No annotation/emotion tracking at the student level** — The `StagnationData` model has `annotation_sentiment` (0.0-1.0), and the normalization service handles it, but there is no corresponding UI element in the mobile contracts where a student can express frustration, confusion, or excitement. The annotation sentiment score has no source. **Fix:** Add a quick emotion check-in widget to the session screen (5 emoji faces, once every 10 questions). Feed this directly into the stagnation detection pipeline and the LLM context. This also provides ground truth for validating the automated behavioral signals.

7. **Methodology switching prompt allows too many switches** — `prompt-templates.py:514-517` The prompt says "Do NOT switch if stagnation score < 0.4" but the MCM `min_attempts_for_switch` in `neo4j-schema.cypher` is as low as 2 (for conceptual -> socratic). A student could theoretically be switched after only 2 failed attempts, which is far too soon to conclude a methodology is failing. With assessment noise (P(slip)=0.10), two consecutive incorrect answers on a known concept happens ~1% of the time purely by chance. **Fix:** Set a floor of `min_attempts_for_switch >= 5` for all MCM entries. Also, the system should require stagnation detection (3 consecutive sessions > 0.7) AND min_attempts exceeded before switching — both conditions, not either.

8. **Knowledge graph visualization is explorer-mode only — no guided path** — `knowledge_graph_widget.dart` — The knowledge graph shows all concepts with zoom/pan and tap-to-inspect, which is great for exploration. But there is no "recommended next concept" highlight, no guided path through the DAG, and no "learning frontier" visualization. Research on learner autonomy (Deci & Ryan 2000, Self-Determination Theory) shows that open exploration works for high-autonomy students but paralyzes low-autonomy students (who just want to be told what to do next). **Fix:** Add a `LearningFrontier` overlay mode that highlights the 2-3 concepts at the frontier of the student's mastery boundary (prerequisites mastered, concept not yet mastered) with a pulsing "Start Here" indicator. Toggle between "Explore" and "Guided" modes.

9. **No Hebrew-specific LLM quality gate for mathematical content** — `acl-interfaces.py:821-832` `HebrewQualityGateError` exists as an exception class, but there is no corresponding quality scoring service or threshold in the contracts. How is the Hebrew quality score computed? What is the threshold? For mathematical Hebrew, common LLM failures include: incorrect gender agreement on mathematical terms, mixing university-level and Bagrut-level terminology, and RTL rendering issues in LaTeX-mixed Hebrew. **Fix:** Define an `IHebrewQualityService` interface that scores LLM Hebrew output on fluency, terminology correctness, and mathematical notation accuracy. Set threshold at 0.80 for tutoring dialogue and 0.90 for formal feedback/evaluation text.

10. **Budget exhaustion gives students a "come back tomorrow" message with no offline fallback** — `cost-tracking.py:229-230` — When the daily 25K output token budget is exhausted, the student hits a hard wall. There is no fallback to pre-generated content, cached exercises, or a degraded mode. A motivated student in the middle of exam prep who hits the budget ceiling at 3pm will be frustrated. **Fix:** When budget is exhausted, fall back to: (a) pre-cached challenge cards that require no LLM, (b) spaced repetition review of previously mastered concepts using pre-generated questions, (c) knowledge graph exploration mode. The student should never be told "come back tomorrow" with no learning option.

### WHAT'S ACTUALLY GOOD

1. **PII stripping on untrusted provider (Kimi) is well-designed** — The `PII` annotation on `StudentContext.student_id` and the `PIIStripper` abstract interface with bidirectional anonymization (strip + restore) is a thoughtful, privacy-by-design approach. Using Pydantic field metadata for automatic PII detection is elegant.

2. **Error taxonomy is pedagogically sound** — The 4-type error classification (procedural, conceptual, motivational, notation) maps directly to established mathematics education research (Radatz 1979, "Error Analysis in Mathematics Education"). Adding behavioral signals (response time, backspaces, answer changes) to the classification context is innovative and well-grounded.

3. **MCM graph structure is the right abstraction** — Using `(ErrorType, ConceptCategory) -> ranked Methodology list` is exactly how Squirrel AI's adaptive engine works, and representing it as a Neo4j graph with confidence scores and sample sizes makes it queryable and auditable. The `min_attempts_for_switch` on each edge is a good safeguard against premature switching.

4. **Methodology switch is routed to Opus (highest-stakes = highest-capability model)** — This is correct prioritization. Methodology switches are infrequent (~2/day) but have the highest impact on learning trajectory. Using Opus for multi-factor reasoning here is justified; the cost is negligible at that frequency.

5. **Cognitive load break screen is non-blocking** — `session_screen.dart:522` "The button is immediately available (timer is a suggestion, not a lock)" — This is exactly right. Mandatory breaks are patronizing and cause frustration. Suggested breaks with an immediate override respect student autonomy while nudging toward healthier study patterns.

6. **Leaderboard is opt-in, class-level, effort-based, anonymized, with hide-self option** — `streak_widget.dart:277-310` — This is one of the most thoughtful leaderboard designs I've reviewed. Every anti-pattern is addressed: opt-in prevents social pressure on non-consenting students, class-level prevents unhealthy school-wide competition, effort-based (not score-based) reduces performance anxiety, anonymization protects identity, and the hide option gives full control. The disclaimer "This leaderboard shows study effort, not scores" is excellent framing.

7. **Prompt templates have few-shot examples with Hebrew math** — The 2-3 few-shot examples per prompt, written in correct Hebrew mathematical terminology with realistic student scenarios, will dramatically improve LLM output quality. The examples cover the full spectrum (low/medium/high mastery, correct/incorrect/motivational) which gives the LLM good calibration anchors.

8. **Partial credit rubric in answer evaluation is research-aligned** — `prompt-templates.py:274-278` — The 60-80% rule for correct method / execution error, 40-60% for right idea / incomplete execution, etc. aligns with Bagrut exam scoring practices and gives students credit for partial understanding, which maintains motivation.

9. **Diagram generation pipeline (batch overnight, CDN-cached, served instantly) solves the latency problem** — LLM-generated diagrams during a session would add 3-10 seconds of latency per question. Pre-generating and caching eliminates this entirely. The DiagramReviewStatus with expert approval flow adds a quality gate.

10. **Student-facing methodology labels hide internal complexity** — `session_screen.dart:397-441` — Translating "spaced_repetition" to "Review Mode / מצב חזרה" and "socratic" to "Guided Discovery / מצב גילוי מודרך" is correct. Students should never see educational jargon; they should see approachable labels that give them a sense of control without overwhelming them with pedagogy theory.
