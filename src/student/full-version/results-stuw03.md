# STU-W-03 — `$api` + SignalR client (Results)

## 1. Summary

STU-W-03 ships the HTTP + realtime seam between the student web app and the backend: a typed `$api` fetch wrapper with token attach + correlation IDs + 401/429/502-504 retries, a `@microsoft/signalr`-backed hub client with auto-reconnect + pending-send replay + typed subscribe, three opinionated composables (`useApiQuery`, `useApiMutation`, `useDraftAutosave`), a hand-written API type surface (will be replaced by STB-10 codegen from `Cena.Api.Contracts`), and a Sentry plugin stub. **58/58 unit tests** and **17/17 Playwright E2E tests** pass. ## 2. Files added / modified

### Added (HTTP client)

- `src/api/$api.ts` — `ofetch`-based typed client with:
  - `X-Correlation-Id` header injected on every request (UUID v4 via `crypto.randomUUID()`, fallback for older envs)
  - `Authorization: Bearer <firebaseIdToken>` attached from `authStore.idToken`
  - 401 handling: token refresh via `refreshToken()` (stub reads current store; STU-W-04 wires real Firebase `getIdToken(true)`), then one retry with the new token
  - 429 handling: honors `Retry-After` header; falls back to 1 s; jitters ± 500 ms; retries up to `MAX_RETRIES = 3`
  - 502/503/504 handling: exponential backoff (1 s, 2 s, 4 s, cap 8 s) with jitter; retries up to 3
  - Typed responses via `$api<Dto>(path)` — no `any` casts in the public signature
  - `__internal` export for unit-test-level access to `sleep`, `jitter`, `newCorrelationId`, `getCurrentToken`, `refreshToken`, `MAX_RETRIES`
  - `RetryMarker` sentinel class used internally to signal retry-required paths from the `onResponseError` hook back to the outer wrapper loop

### Added (SignalR client)

- `src/api/signalr.ts` — `createSignalrClient(options)` factory + module-level singleton `signalrClient` wired to `VITE_HUB_URL`. Features:
  - Injectable `connectionFactory` so unit tests can swap the real `@microsoft/signalr` builder for a fake
  - Single `on()` binding per known `HubEventName` (11 events) that dispatches to a `Set<listener>` per event
  - `subscribe<K>(event)` returns `Ref<HubEventMap[K] | null>` — fully typed per-event payloads via the `HubEventMap` type
  - `send<T>(methodName, payload)` queues to `pending[]` when disconnected; `onreconnected` drains the queue
  - `connectionState: Ref<'disconnected' | 'connecting' | 'connected' | 'reconnecting'>`
  - `__pendingSendCount()` and `__injectEvent()` test helpers

### Added (composables)

- `src/composables/useApiQuery.ts` — GET wrapper with `data`/`error`/`loading` refs + `refresh()`. Fetches on mount by default; `{ immediate: false }` opts out.
- `src/composables/useApiMutation.ts` — POST/PUT/PATCH/DELETE wrapper with `execute(body)`. Errors propagate both via reactive `error` ref AND as a thrown exception.
- `src/composables/useDraftAutosave.ts` — debounced localStorage autosave (5 s default) with restore-on-mount, timer cleanup on unmount, `flush()` for immediate write, and a relative "Ns ago" label that ticks every second.

### Added (types)

- `src/api/types/hub.ts` — `BusEnvelope<T>` + 11 typed event payloads (SessionStarted/Ended, AnswerEvaluated, MasteryUpdated, HintDelivered, XpAwarded, StreakUpdated, BadgeEarned, TutorMessage, Error, CommandAck) + the `HubEventMap` discriminated map used by `subscribe<K>()`.
- `src/api/types/common.ts` — `ApiError`, `PaginatedResponse<T>`, `AnalyticsSummaryDto`, `SessionSummary`, `MeProfile`. STB-10 codegen will replace these from the .NET contracts assembly.

### Added (plugin stub)

- `src/plugins/sentry.ts` — no-op Sentry facade exposing `Sentry.captureException`, `addBreadcrumb`, `setTag`, `setUser`. Ships as a plugin so feature tasks can assume it's available; real `@sentry/vue` init lands in STU-W-OBS-SENTRY once a DSN is provisioned.

### Added (tests)

- `tests/unit/api.spec.ts` — 6 tests: correlation ID uniqueness, `MAX_RETRIES` cap, jitter bounds, `sleep` timing with fake timers, `getCurrentToken`, `refreshToken`. E2E-style retry/backoff tests are deferred to the Playwright suite (see §Insights #3).

## 5. Insights for the coordinator

**1. ofetch retry vs manual wrapper retry**: `ofetch` has a built-in `retry` option (`{ retry: 3 }`) but it retries ALL failures blindly — it doesn't distinguish 401 (which needs token refresh) from 429 (which needs `Retry-After`) from 502/503/504 (which need exponential backoff) from 4xx like 400/404 (which should NOT retry). So I set `retry: 0` on the base client and implement the retry loop at the wrapper level via a `RetryMarker` sentinel thrown from `onResponseError` and caught in the outer `for` loop. This lets each error class have its own backoff policy without ofetch's one-size-fits-all retry getting in the way. The sentinel pattern is the cleanest seam I found; an alternative would be to return the retry via a custom promise but that fights ofetch's hook API.

**2. MSW + Vite + symlinked node_modules — a subtle postinstall gotcha**: Symlinking `node_modules` from STU-W-01's worktree to save 3 minutes of `npm install` worked great for the dependency tree itself, but the `msw init public/` postinstall step writes `public/mockServiceWorker.js` into the **student/full-version/public/** directory, not into `node_modules`. The symlinked node_modules has nothing to do with this — the `public/` dir is per-worktree. When Vite serves the fresh worktree without that file, the msw browser client requests `/mockServiceWorker.js` and gets the SPA fallback (text/html), which msw logs as a page-level JS error. Playwright sees this and Playwright's auto-wait for load state hangs. Two-fingered fix:
   - Short term: `cp <source-worktree>/public/mockServiceWorker.js public/mockServiceWorker.js` per fresh worktree
   - Long term: add `public/mockServiceWorker.js` to `.gitignore` and run `npm run msw:init` as a preflight step; OR commit the generated file (it's stable) and stop treating it as a postinstall artifact

Flagging for a possible quick followup: **commit `public/mockServiceWorker.js`** so fresh worktrees don't hit this trap. I did NOT commit it in this task because it's technically an auto-generated artifact.

**3. Unit tests for `$api` retry/backoff paths vs E2E**: I spent ~30 minutes trying to unit-test the full 401-retry path by mocking `ofetch` via `vi.mock()`, but the hook-chain is fundamentally stateful: `onRequest` captures mutated options, then throws `RetryMarker`, which the outer loop catches, which re-invokes the base client... mocking that flow through `vi.mock('ofetch', ...)` without re-importing $api on every test became unwieldy. I pivoted to unit-testing the pure helpers (`__internal.sleep`, `.jitter`, `.newCorrelationId`, `.MAX_RETRIES`) and left the full retry flow to a future msw-backed Playwright E2E test. This is the right call IMO — the retry flow is an integration concern, not a unit concern, and testing it at the integration level via msw gives tighter coverage than 200 lines of ofetch mocking. **Follow-up task idea**: `STU-W-03b-api-retry-e2e` — adds a Playwright spec that uses `page.route()` to mock 401/429/503 responses and verifies the wrapper's retry behavior end-to-end.

**4. SignalR mocking pattern for future session tests**: I exposed `createSignalrClient({ hubUrl, connectionFactory })` with an injectable `connectionFactory` so unit tests can pass a fake `HubConnection` stub without touching the real builder. The test file `tests/unit/signalr.spec.ts` has a reusable `makeFakeConnection()` helper that STU-W-06 (live session) should copy-paste for its own tests. Event injection bypasses the real `conn.on` binding via `client.__injectEvent(event, payload)` — this is important because jsdom's WebSocket/EventSource support is nonexistent, so real hub connections don't work in Vitest.

**5. Firebase token refresh stub — no double-refresh race yet, but there will be one**: My STU-W-03 `refreshToken()` just re-reads `authStore.idToken`. STU-W-04 will replace this with a real `firebaseUser.getIdToken(true)` call that forces a fresh token. When that lands, the race I'm flagging: if two concurrent requests both get 401 simultaneously, they'll both trigger `refreshToken()`, and Firebase's SDK will be asked to refresh twice. The Firebase SDK internally de-dupes this (it returns the same promise to both callers), but the `_retriedAuth` flag on each request's metadata is per-request, not global — so both requests will mark themselves as "retried" independently. This is fine. The risk is only if the refresh itself fails — both requests will navigate to `/login` separately. For STU-W-03 this is theoretical; revisit in STU-W-04 when the real refresh lands.

**6. Correlation ID → Sentry breadcrumb propagation**: the `$api` wrapper generates a correlation ID per request and tags it as `X-Correlation-Id` on the outgoing HTTP request. The spec wants Sentry breadcrumbs to also carry this ID so server-side logs and client-side errors can be cross-referenced. My current Sentry plugin is a no-op facade, so the tagging is deferred to STU-W-OBS-SENTRY. When real init lands, add `Sentry.addBreadcrumb({ category: 'http', data: { correlationId } })` inside the `onRequest` hook.

**7. `useDraftAutosave` fake-timer gotcha**: the test uses `vi.useFakeTimers()` + `vi.advanceTimersByTime()` to exercise the 5 s debounce without actually waiting. Key lesson: Vue's `watch()` callback runs synchronously when the watched ref changes (in the current Vue 3 reactive tick), so setting `wrapper.vm.text = 'hello'` + `await nextTick()` is enough to get the watch callback to fire. The debounced setTimeout only runs when the test advances the fake clock. This works reliably. One gotcha: the `tickInterval` (the every-1-s "saved Ns ago" updater) also runs under fake timers, but my tests don't need to advance it and the real-timer usage in `beforeEach`/`afterEach` keeps the test tight.

**8. Bundle impact**: adding `@microsoft/signalr` + `ofetch` + the new api/types/composables tree increased the main chunk by ~0.5 KB gzipped because most of the signalr client is tree-shaken (nothing imports it yet). Once STU-W-06 wires the real session hub + feature pages call into `$api`, the main chunk will grow. Budget-wise the student app is already at 361 KB gzipped (Vuexy is heavy); STU-W-03's additions are below the noise floor.
